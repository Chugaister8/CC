# 3.4. Автентифікація в ОС

Автентифікація — перший рубіж захисту будь-якої системи. Все, що описано в попередньому розділі про права доступу, починається з одного питання: «а ти взагалі хто?» Якщо відповідь на це питання можна підробити або обійти, вся подальша система прав — просто декорація. Для ОС питання автентифікації охоплює не лише пароль при вході, а й те, як ці паролі зберігаються, як перевіряються, які альтернативи є і де найчастіше виникають помилки.

> 📖 Ключові терміни — у [глосарії модуля](00-glosariy.md).

## Чинники автентифікації

Класична класифікація чинників:

- **Знання (Something you know):** пароль, PIN, секретне запитання.
- **Володіння (Something you have):** фізичний токен (YubiKey), смартфон (TOTP-застосунок), смарт-карта.
- **Ознака (Something you are):** біометрія — відбиток пальця, обличчя, райдужна оболонка.
- **Місцезнаходження (Somewhere you are):** IP-адреса, геолокація — додатковий чинник, але не самостійний.

**MFA (Multi-Factor Authentication)** — автентифікація з двома або більше різними чинниками. Поєднання пароля і TOTP-коду — це MFA; поєднання двох паролів — ні (обидва з одного чинника «знання»).

## Зберігання паролів в ОС: як це реально працює

ОС ніколи не зберігають паролі у відкритому вигляді — лише їхні хеші. При автентифікації система хешує введений пароль і порівнює з збереженим хешем.

### Windows: SAM і NTLM

**SAM (Security Account Manager)** — база даних локальних облікових записів Windows, зберігається у `C:\Windows\System32\config\SAM`. Захищена ключем SYSKEY і недоступна під час роботи ОС навіть для адміністратора.

Формат збереженого хешу — **NT hash** (MD4 від Unicode-представлення пароля):

```
Alice:1001:NT_HASH_HERE:::
```

**NTLM (NT LAN Manager)** — протокол автентифікації Windows, заснований на NT hash. Вразливий до атак Pass-the-Hash: зловмисник, що отримав NT hash (наприклад, через Mimikatz або дамп SAM), може автентифікуватись без знання пароля.

**Kerberos** — безпечніший протокол автентифікації для доменного середовища Active Directory, заснований на тікетах і не передає хеш пароля по мережі.

### Linux: /etc/shadow

Паролі Linux зберігаються у `/etc/shadow` (читається лише root):

```
alice:$6$rounds=5000$salt$hash_here:19000:0:99999:7:::
```

Перша частина після `:` — алгоритм хешування:
- `$1$` — MD5 (застарілий, не використовувати)
- `$5$` — SHA-256
- `$6$` — SHA-512 (стандарт)
- `$y$` — yescrypt (сучасний, рекомендований)

Разом із хешем зберігається **salt** — випадкова рядок, що додається до пароля перед хешуванням. Це унеможливлює використання передобчислених таблиць (Rainbow Tables): однаковий пароль з різними salt дає різні хеші.

### Атаки на паролі

| Тип атаки | Механізм | Захист |
|---|---|---|
| **Brute force** | Перебір усіх можливих комбінацій | Довгий пароль, блокування після N спроб |
| **Dictionary attack** | Перебір за списком відомих паролів | Унікальний пароль, не зі словника |
| **Rainbow Table** | Пошук хешу в передобчисленій таблиці | Сильний salt |
| **Pass-the-Hash** | Використання хешу замість пароля | Захист облікових записів, Credential Guard |
| **Credential Stuffing** | Використання витоків паролів з інших сервісів | Унікальний пароль для кожного сервісу |

## Парольні політики

**Застарілий підхід (до 2017 року):** мінімум 8 символів, великі/маленькі літери, цифри, символи, обов'язкова зміна кожні 90 днів.

**Проблема:** людська психологія протидіє цим вимогам: `Password1!` → `Password2!` → `Password3!`. Регулярна зміна призводить до передбачуваних патернів, а не до справжньої безпеки.

**Сучасний підхід (NIST SP 800-63B, 2017+):**
- Мінімальна довжина — 8+ символів; рекомендована — 15+.
- Перевіряти пароль на наявність у відомих витоках (Have I Been Pwned API).
- **Не вимагати** обов'язкову регулярну зміну без підстав.
- **Не вимагати** складності символів — краще довжина.
- Блокувати відомо слабкі паролі (`password`, `123456`).
- Довгі passphrase (`КіткаЇстьМолоко2026!`) — безпечніші і легші для пам'яті.

### Налаштування парольної політики

**Linux (`/etc/login.defs`):**
```bash
PASS_MIN_LEN    12        # мінімальна довжина
PASS_MAX_DAYS   365       # максимальний термін дії
PASS_MIN_DAYS   1         # мінімальний час між змінами
PASS_WARN_AGE   14        # попередження за N днів
```

**Linux PAM (Pluggable Authentication Modules)** — гнучкий фреймворк автентифікації; `/etc/pam.d/common-password`:
```
password requisite pam_pwquality.so retry=3 minlen=12 difok=3 reject_username
password required pam_unix.so sha512 shadow nullok remember=5
```

**Windows (Local Security Policy або Group Policy):**
```
Мінімальна довжина пароля: 12
Максимальний термін дії: 365 днів
Ведення журналу паролів: 5 останніх
Складність пароля: вимагати
Блокування після: 5 невдалих спроб → блокування на 30 хвилин
```

## Windows Hello і сучасна автентифікація

**Windows Hello** — підсистема автентифікації Windows 10/11, що підтримує:
- **PIN** — не є звичайним паролем: прив'язаний до конкретного пристрою і апаратно захищений TPM. Знання PIN без фізичного доступу до пристрою марне.
- **Відбиток пальця** — зберігається в TPM, не передається нікуди.
- **Розпізнавання обличчя** — через ІЧ-камеру (Windows Hello Face).

Технічна основа Windows Hello — **FIDO2/WebAuthn**: замість пароля генерується пара ключів (RSA/EC), приватний ключ зберігається в TPM пристрою. Навіть фішингова атака, що захопила «пароль», не отримає приватний ключ.

## Passkeys: паролі майбутнього

**Passkeys** — розвиток FIDO2, підтримуваний Apple, Google, Microsoft. Замість пароля при реєстрації генерується пара ключів:
- Приватний ключ — у захищеному сховищі пристрою (TEE, Secure Enclave, TPM).
- Публічний ключ — на сервері.

При вході пристрій підписує запит сервера приватним ключем. Пароль не існує — нема чого вкрасти, нема чого зафішити. Passkeys — перший реальний масовий механізм автентифікації, що повністю усуває ризик credential stuffing і фішингу паролів.

## MFA на рівні ОС

**TOTP (Time-based One-Time Password)** — одноразовий код, що змінюється кожні 30 секунд (Google Authenticator, Authy, Microsoft Authenticator). Заснований на HMAC-SHA1 від поточного часу і спільного секрету.

### Підключення MFA до Linux (SSH і локальний вхід)

```bash
# Встановити Google Authenticator PAM
sudo apt install libpam-google-authenticator

# Налаштувати для поточного користувача
google-authenticator

# Додати до /etc/pam.d/sshd:
auth required pam_google_authenticator.so

# В /etc/ssh/sshd_config:
ChallengeResponseAuthentication yes
AuthenticationMethods publickey,keyboard-interactive
```

### MFA для Windows (локальна система)

Для локальних Windows-машин поза Active Directory MFA реалізується переважно через сторонні рішення (Duo Security, WinAuth) або Windows Hello (що є фактором «ознака/володіння»).

## SSH-ключі: правильна автентифікація для Linux

Автентифікація за паролем у SSH — вразлива до брутфорсу. Ключі — значно надійніший підхід.

```bash
# Генерація пари ключів (Ed25519 — сучасний стандарт)
ssh-keygen -t ed25519 -C "your@email.com"
# Зберігає: ~/.ssh/id_ed25519 (приватний) і ~/.ssh/id_ed25519.pub (публічний)

# Копіювання публічного ключа на сервер
ssh-copy-id user@server

# Або вручну: додати вміст .pub до ~/.ssh/authorized_keys на сервері
```

**Після налаштування ключів — вимкнути автентифікацію за паролем** (у `/etc/ssh/sshd_config`):
```
PasswordAuthentication no
PubkeyAuthentication yes
```

## Типові помилки автентифікації на рівні ОС

| Помилка | Наслідок | Виправлення |
|---|---|---|
| Не змінено заводський пароль адміністратора | Тривіальний злом | Змінити негайно |
| Однаковий пароль локального адміна на всіх машинах | Lateral movement при компрометації одного | Windows LAPS (Local Admin Password Solution) |
| Вимкнена блокування після N спроб | Брутфорс без обмежень | Увімкнути Account Lockout Policy |
| Активний `root`-вхід по SSH | Брутфорс напряму на root | `PermitRootLogin no` в SSH-конфігу |
| Порожній або слабкий пароль | Тривіальний злом | Складний уніальний пароль або ключ |
| MFA відсутній для критичних систем | Пароль = єдиний захист | Увімкнути MFA |

## Міні-вправа

**Linux:**
```bash
# Перевірити облікові записи з порожніми паролями
sudo awk -F: '($2 == "" ) { print $1 " має порожній пароль!" }' /etc/shadow

# Переглянути дату останньої зміни пароля
sudo chage -l your_username

# Перевірити, чи вимкнено вхід під root по SSH
grep -i "PermitRootLogin" /etc/ssh/sshd_config
```

**Windows (PowerShell):**
```powershell
# Знайти облікові записи з незакінчуючимся паролем
Get-LocalUser | Where-Object {$_.PasswordNeverExpires -eq $true} | Select-Object Name

# Знайти активні облікові записи без пароля
Get-LocalUser | Where-Object {$_.PasswordRequired -eq $false -and $_.Enabled -eq $true}
```
