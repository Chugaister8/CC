# 3.3. Користувачі, групи та права доступу

У попередньому розділі ми з'ясували, що ОС має апаратно реалізовані межі між рівнями привілеїв. Але ці межі — лише каркас. Реальний розподіл «хто що може» всередині одного рівня визначається системою прав доступу: хто власник файлу, яка у нього група, які права є у решти. Права доступу — це мова, якою ОС описує дозволи; розуміти цю мову — означає вміти і налаштовувати захист, і помічати, де він порушений. Розуміти цю мову — означає вміти і налаштовувати захист, і аналізувати, де він порушений. Більшість реальних інцидентів включають або надмірно широкі права (користувач чи процес мав доступ, якого не мав би мати), або неправильно виставлені права на критичні файли. Обидві проблеми вирішуються свідомим управлінням правами доступу.

> 📖 Ключові терміни — у [глосарії модуля](00-glosariy.md).

## Моделі контролю доступу

Існує кілька фундаментальних моделей контролю доступу:

- **DAC (Discretionary Access Control)** — власник ресурсу сам вирішує, кому надати доступ. Стандартна модель NTFS і Linux. Гнучка, але залежить від свідомості власника.
- **MAC (Mandatory Access Control)** — система примусово накладає мітки безпеки на об'єкти і суб'єкти; навіть власник не може надати доступ в обхід правил. AppArmor і SELinux у Linux; частково Windows Integrity Levels.
- **RBAC (Role-Based Access Control)** — права призначаються ролям (посадам), а не окремим користувачам; користувач отримує права через членство в ролі. Стандарт для корпоративних систем.

## Linux: права доступу до файлів

### Базова модель

Кожен файл і директорія в Linux мають три набори прав і три категорії суб'єктів:

```
-rwxr-xr-- 1 alice developers 4096 Jan 15 10:00 script.sh
│││││││││└─ інші (others): r-- (читання)
││││││└──── група (group): r-x (читання + виконання)
│││└──────── власник (user): rwx (читання + запис + виконання)
││└───────── тип файлу: - (звичайний), d (директорія), l (посилання)
```

**Числовий запис:**

| Право | Символ | Число для файлу | Число для директорії |
|---|---|---|---|
| Read | r | читання вмісту | перелік файлів у директорії |
| Write | w | зміна вмісту | створення/видалення файлів |
| Execute | x | запуск як програми | вхід у директорію (`cd`) |

```bash
# Приклади: числові права
chmod 755 script.sh   # rwxr-xr-x: власник — все, група та інші — читання+виконання
chmod 644 config.txt  # rw-r--r--: власник — читання+запис, решта — лише читання
chmod 600 id_rsa      # rw-------: лише власник, читання+запис (закритий ключ SSH)
chmod 700 ~/.ssh      # rwx------: лише власник має доступ до директорії
```

### Спеціальні біти

| Біт | Назва | Числовий префікс | Ефект |
|---|---|---|---|
| **SUID** | Set User ID | 4xxx | Файл виконується від імені власника файлу, а не того хто запускає |
| **SGID** | Set Group ID | 2xxx | Файл виконується від імені групи файлу; для директорій — нові файли успадковують групу |
| **Sticky bit** | — | 1xxx | У директорії користувач може видаляти лише власні файли (наприклад, `/tmp`) |

**Чому SUID — ризик безпеки:** якщо SUID-файл має вразливість, зловмисник може виконати довільний код від імені власника (часто root). Тому кількість SUID-файлів у системі має бути мінімальною.

```bash
# Знайти всі SUID-файли в системі (можлива ескалація привілеїв)
find / -perm -4000 -type f 2>/dev/null

# Знайти файли з надміру широкими правами (доступні всім на запис)
find / -perm -o+w -not -path "/proc/*" -type f 2>/dev/null | head -20
```

### Команди управління правами і власністю

```bash
# Змінити власника
chown alice:developers file.txt

# Рекурсивно змінити власника директорії
chown -R www-data:www-data /var/www/html

# Переглянути права у зручному форматі
stat file.txt

# Налаштувати маску за замовчуванням (umask 022 = нові файли 644, директорії 755)
umask 027   # нові файли 640, директорії 750 — більш суворо
```

### ACL в Linux

Стандартна модель прав (user/group/other) недостатня, коли потрібно надати права конкретному користувачу, що не є власником і не входить у групу файлу. Для цього є **POSIX ACL**:

```bash
# Надати користувачу bob права на читання файлу
setfacl -m u:bob:r file.txt

# Переглянути ACL
getfacl file.txt

# Видалити ACL
setfacl -x u:bob file.txt
```

### sudo: контрольоване підвищення привілеїв

**sudo** (superuser do) — стандартний механізм виконання команд з правами root без входу як root.

Конфігурація у `/etc/sudoers` (завжди редагувати через `visudo`):

```bash
# Надати alice право запускати будь-яку команду від root
alice ALL=(ALL:ALL) ALL

# Надати bob право перезапускати nginx без пароля
bob ALL=(root) NOPASSWD: /usr/bin/systemctl restart nginx

# Надати групу devops право перезапускати сервіси
%devops ALL=(root) /usr/bin/systemctl restart *

# Заборонити певні небезпечні команди (погана практика — краще дозволяти явно)
```

**Безпекові принципи sudo:**
- Ніколи не давати `ALL=(ALL:ALL) ALL` без необхідності.
- Уникати `NOPASSWD` для потужних команд.
- Не включати до sudoers інтерпретатори (`python`, `bash`, `vim`) — через них тривіально отримати shell.
- Логи sudo за замовчуванням пишуться в `/var/log/auth.log`.

## Windows: права доступу NTFS та UAC

### NTFS-дозволи

Windows використовує ACL на основі **SID (Security Identifier)** — унікальних ідентифікаторів користувачів і груп. Дозволи накопичуються від різних груп і можуть явно заперечуватись (**Deny**).

Основні типи дозволів NTFS:

| Дозвіл | Файл | Папка |
|---|---|---|
| Full Control | Усе | Усе, включно зі зміною дозволів |
| Modify | Читання, запис, видалення | Читання, запис, видалення вмісту |
| Read & Execute | Читання і запуск | Читання і запуск файлів у папці |
| Read | Лише читання | Перелік вмісту |
| Write | Запис і створення | Створення файлів і папок |

```powershell
# Переглянути NTFS-дозволи на файл
Get-Acl "C:\important\file.txt" | Format-List

# Переглянути дозволи у зрозумілому вигляді
icacls "C:\important\file.txt"

# Призначити дозволи
icacls "C:\data\" /grant "Alice:(OI)(CI)M" /T
# (OI) — Object Inherit, (CI) — Container Inherit, M — Modify
```

### UAC: User Account Control

**UAC** — механізм Windows, що запитує підтвердження або облікові дані адміністратора при спробі виконати дію, що потребує підвищених привілеїв. Навіть якщо поточний користувач є членом групи Administrators, стандартні операції виконуються з обмеженим токеном (filtered token). Лише при явному підтвердженні UAC-запиту видається токен з повними привілеями.

**Рівні UAC (у порядку від найменш до найбільш безпечного):**
- Ніколи не повідомляти — UAC вимкнено. Критично небезпечно.
- Повідомляти лише при змінах застосунками (без затемнення екрана) — стандартний за замовчуванням. Вразливий до деяких обходів.
- Повідомляти лише при змінах застосунками (із затемненням) — краще.
- Завжди повідомляти — максимальний захист, але найбільш «нав'язливий».

**Атаки на UAC (UAC bypass)** — існують техніки, що дозволяють обійти UAC без запиту (через підписані бінарники, автоавтопідвищення тощо). Жоден рівень UAC не є абсолютним захистом від зловмисника з уже запущеним кодом.

### Windows Integrity Levels

Окрім DAC, Windows має **Mandatory Integrity Control (MIC)** — мітки цілісності:

| Рівень | Де використовується |
|---|---|
| System | Ядро, SYSTEM-процеси |
| High | Процеси з підвищеними привілеями (UAC) |
| Medium | Звичайні процеси |
| Low | Браузери (захищений режим), завантажені файли |
| Untrusted | Процеси з дуже обмеженим доступом |

Процес з нижчим рівнем цілісності не може записувати у об'єкти з вищим рівнем навіть якщо DAC-дозволи це допускають.

## Групова модель і принцип найменших привілеїв на практиці

**Типові помилки управління правами:**

1. **«Всі в групі Administrators»** — класика корпоративного ІТ. Якщо заражено один комп'ютер і всі мають однакові облікові дані адміністратора — зловмисник отримує весь домен.

2. **Спільні облікові записи** — `admin/admin` для кількох людей: неможливо відстежити, хто що зробив.

3. **Сервісні облікові записи з правами адміністратора** — веб-сервер, що виконується від `SYSTEM` або `root`, після компрометації дає зловмиснику повний контроль.

4. **Надмірні права на файли/папки** — директорія бази даних, доступна на запис для `Everyone` або `Other`.

**Правильний підхід:**
- Для кожної служби — окремий сервісний обліковий запис з мінімально необхідними правами.
- Групи управляють правами; користувачі отримують права лише через членство в групах.
- Аудит прав проводиться регулярно (особливо після змін в організації).
- При звільненні співробітника — негайна деактивація облікового запису і зміна паролів спільних ресурсів.

## Міні-вправа

**Linux:**
```bash
# Перевірити всі SUID-файли в системі
find / -perm -4000 -type f 2>/dev/null
# Порівняйте список з еталонним для вашого дистрибутиву.
# Будь-який незнайомий файл — підозрілий.

# Перевірити свої поточні sudo-привілеї
sudo -l
```

**Windows (PowerShell від адміністратора):**
```powershell
# Переглянути членів групи Administrators на локальній машині
Get-LocalGroupMember -Group "Administrators"
# Чи є тут облікові записи, яким не потрібні адмін-права?

# Переглянути локальних користувачів
Get-LocalUser | Select-Object Name, Enabled, LastLogon, PasswordLastSet
```
