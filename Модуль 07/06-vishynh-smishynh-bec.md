# 7.6. Вішинг, смішинг, BEC і квішинг

Поки фільтри email-шлюзів стають розумнішими, зловмисники розширюють канали: телефонний дзвінок, SMS, QR-код, месенджер. Кожен канал має власні психологічні особливості і технічні обмеження фільтрації. Телефонний дзвінок «від банку» спрацьовує тому, що більшість людей ніколи не стикалась з підробленими дзвінками і не мають захисних рефлексів, які виробились щодо підозрілих листів.

> 📖 Ключові терміни — у [глосарії модуля](00-glosariy.md).

## Вішинг (Voice Phishing)

Вішинг — соціальна інженерія через голосовий дзвінок. На відміну від email-фішингу, людина спілкується в реальному часі, що посилює психологічний тиск і зменшує час на роздуми.

**Типові сценарії:**

**«Ваш акаунт зламано»** — найпоширеніший шаблон:
```
«Доброго дня, це служба безпеки ПриватБанку. Ми виявили підозрілу транзакцію
 на вашій картці. Для захисту коштів мені необхідно верифікувати ваші дані.
 Назвіть останні 4 цифри картки і CVV...»
```

**«Техпідтримка Microsoft»:**
```
«This is Microsoft Technical Support. Your computer has been sending us
 error reports and is infected with a virus. Please allow me to connect
 remotely to fix the issue...»
→ Встановлення RAT (AnyDesk, TeamViewer, Quick Assist)
```

**«Повернення переплати» (Refund Scam):**
```
«Ви переплатили за підписку. Нам потрібно повернути вам кошти.
 Відкрийте ваш банківський застосунок і продиктуйте код підтвердження...»
→ Насправді: переказ коштів зі скомпрометованого акаунту
```

**Захист:**
- Якщо «банк» телефонує першим → покладіть слухавку і зателефонуйте за офіційним номером з картки або сайту.
- Банки ніколи не просять CVV, OTP, PIN по телефону.
- Caller ID підробляється (spoofed) — «номер банку» в визначнику не є гарантією.

**CLI spoofing:** Зловмисники використовують VoIP-сервіси, що дозволяють вказати довільний Caller ID. Легальна послуга (для корпоративних АТС) використовується незаконно.

## Смішинг (SMS Phishing)

SMS-фішинг має вищий відсоток відкриття, ніж email (~98% vs ~20%), і менше захисних фільтрів.

**Типові шаблони:**

```
[NOVA POSHTA] Ваша посилка #UA12345 чекає підтвердження адреси.
Перейдіть за посиланням: bit.ly/np-confirm-ua

Повідомлення від ПриватБанку: Ваш рахунок буде заблоковано.
Деталі: privat24-secure.xyz/verify

УВАГА! Ваш акаунт Дія підозрілий. Верифікуйте: diia-gov-ua.ru/login
```

**Характерні ознаки:**
- Скорочені URL (bit.ly, tinyurl) — приховують справжній домен.
- Домени-імітатори: `nova-poshta-ua.info`, `diia-gov-ua.ru`.
- Підозрілі розширення доменів (`.ru`, `.info` для українських сервісів).
- Граматичні помилки або невластивий стиль.
- Посилання, що «протікають» через перехоплення SMS-OTP.

**SMS OTP Interception:** навіть якщо жертва просто відкрила фішинговий сайт і ввела номер телефону, зловмисник може ініціювати скидання пароля і перехопити OTP-код через підроблену сторінку підтвердження.

## BEC: Business Email Compromise

**BEC** — цільова атака на організації з метою фінансового шахрайства або ексфільтрації даних через компрометований або підроблений корпоративний email.

**FBI IC3 статистика:** BEC-збитки перевищили $50 млрд (2013–2023) — найбільша категорія кіберзбитків.

**Типи BEC:**

| Тип | Сценарій | Ціль |
|---|---|---|
| **CEO Fraud** | Email «від CEO» до бухгалтера з терміновим переказом | Фінансист/бухгалтерія |
| **Vendor Impersonation** | Підроблений рахунок від існуючого постачальника зі «зміненими» реквізитами | Бухгалтерія |
| **HR Impersonation** | «Оновлення зарплатних реквізитів» від HR | Payroll |
| **Attorney Impersonation** | «Конфіденційне юридичне питання» | Керівництво |
| **Data Theft** | Запит W-2 форм або списку співробітників | HR |

**Реальний приклад:**
```
From: ceo@company.com.secure-mail.net  ← не справжній домен
To: accountant@company.com
Subject: Терміново: конфіденційне

Ірино, мені потрібен терміновий переказ 85 000 грн нашому новому партнеру.
Деталі рахунку у вкладенні. Прошу не обговорювати з іншими — це конфіденційна угода.
Я на нараді, відповідати не можу. Виконайте до 16:00.

З повагою,
Олексій Шевченко
Генеральний директор
```

**Техніки BEC:**

1. **Domain Spoofing** — підробка домену відправника (але SPF/DKIM перевіряє).
2. **Lookalike Domain** — `cornpany.com` (rn замість m) — реєструють схожий домен.
3. **Account Takeover** — реальний аккаунт CEO чи партнера зламано.
4. **Thread Hijacking** — відповідь на існуючий email-ланцюжок зі скомпрометованого або підробленого акаунту.

**Захист від BEC:**
- Чітка процедура: фінансові перекази понад певну суму — обов'язкове телефонне підтвердження за відомим номером.
- DMARC з `p=reject` — блокує більшість domain spoofing.
- Навчання фінансових та HR-команд розпізнавати BEC.
- Двостороннє підтвердження для нових банківських реквізитів постачальників.

## Квішинг (QR Code Phishing)

**Квішинг** — фішинг через QR-коди. Набув поширення після 2022 року, коли більшість email-сканерів навчились аналізувати URL, але не «бачать» URL всередині QR-коду.

```
Типові вектори:
- QR-код в PDF або зображенні в email → обходить URL-сканер
- Фізичний QR-код поверх легітимного (парковки, ресторани, кіоски)
- «Отримай документ»: email з інструкцією «відскануйте QR для завантаження»

Ціль: AITM фішинг (Adversary-in-the-Middle) — перехоплення сесійних токенів
після Microsoft 365 або Google Workspace автентифікації
```

**Захист:**
- Email-шлюзи з QR-декодуванням (Microsoft Defender for Office 365 у 2023 додав цю функцію).
- Навчання: перед скануванням QR перевіряйте контекст (хто надіслав, звідки фізичний код).
- Мобільні браузери: перевіряйте попередній перегляд URL перед переходом.

## Deepfake Vishing

Найновіший вектор: AI-синтез голосу або відео для переконливого импersonation.

**Реальний кейс (2024, Гонконг):** Фінансовий директор компанії взяв участь у відеоконференції з «колегами» (включаючи «CFO» і «директорів»). Всі учасники були deepfake-генерованими. Результат: переказ $25 мільйонів.

**Голосовий deepfake** дозволяє клонувати голос за 10–30 секундами аудіо (публічні виступи, відео, подкасти). Зловмисник телефонує і «розмовляє» голосом CEO або родича.

**Захист:**
- Кодові слова або «секретні фрази» для підтвердження особи в критичних ситуаціях.
- Незвичні прохання завжди перевіряти через альтернативний канал.
- Скептицизм до «відеодзвінків», де учасники відмовляються відповідати на довільні запитання або їх міміка виглядає неприродно.

## Міні-вправа

Перевірте захист вашої організації від BEC:

```python
# Перевірка налаштувань email-домену
import dns.resolver  # pip install dnspython

def check_email_security(domain: str) -> None:
    print(f"\n=== Email Security: {domain} ===")

    # SPF
    try:
        spf = dns.resolver.resolve(domain, 'TXT')
        for r in spf:
            txt = r.to_text()
            if 'v=spf1' in txt:
                print(f"✅ SPF: {txt[:80]}...")
                break
    except Exception:
        print("❌ SPF: не знайдено")

    # DMARC
    try:
        dmarc = dns.resolver.resolve(f'_dmarc.{domain}', 'TXT')
        for r in dmarc:
            txt = r.to_text()
            if 'v=DMARC1' in txt:
                policy = 'reject' if 'p=reject' in txt else ('quarantine' if 'p=quarantine' in txt else 'none ⚠️')
                print(f"{'✅' if policy != 'none ⚠️' else '⚠️ '} DMARC: p={policy}")
                break
    except Exception:
        print("❌ DMARC: не знайдено")

    # MX
    try:
        mx = dns.resolver.resolve(domain, 'MX')
        print(f"ℹ️  MX: {[str(r.exchange) for r in mx]}")
    except Exception:
        print("❌ MX: не знайдено")

check_email_security('your-company.com')  # замініть на ваш домен
```
