# 12.3. Сканування вразливостей

## Навіщо автоматизація, якщо є пентест

Логічне запитання: якщо розділ 12.7 описує тестування на проникнення людьми-експертами, навіщо окремо сканування? Відповідь — у масштабі й частоті. Пентест — це глибокий, ручний, дорогий аналіз кількох критичних систем раз на квартал чи рік. Сканування — автоматизований, дешевий, широкий прохід по **всій** інфраструктурі щотижня чи навіть щодня. Це не конкуруючі, а взаємодоповнюючі практики: сканер знаходить 95% відомих, каталогізованих проблем швидко й дешево; пентестер знаходить складні ланцюжки атак і логічні вади бізнес-процесу, які жоден автоматичний інструмент не розпізнає.

## Типи сканерів за об'єктом

- **Network/Infrastructure scanners** (Nessus, OpenVAS, Qualys) — виявляють відкриті порти, застарілі версії служб, відсутні патчі ОС, слабкі конфігурації протоколів (наприклад, застарілі шифри TLS).
- **Web application scanners** (Burp Suite, OWASP ZAP, Acunetix) — спеціалізуються на вебвразливостях з Модуля 06 (SQLi, XSS, IDOR), взаємодіючи з застосунком через HTTP так само, як реальний користувач чи атакувальник.
- **Container/image scanners** (Trivy, Grype, Clair) — аналізують шари Docker-образів на відомі CVE в системних бібліотеках і залежностях, ще до запуску контейнера в продакшені — інтегруються в CI/CD як security gate (див. Модуль 06, API Security).
- **Cloud/Configuration scanners (CSPM)** (Prowler, ScoutSuite, вбудовані AWS Security Hub / Azure Defender) — перевіряють не CVE, а конфігураційні помилки хмарної інфраструктури: публічні S3-бакети, надмірні IAM-права, відсутнє шифрування дисків.

## Автентифіковане проти неавтентифікованого сканування

Це найважливіша дихотомія в цьому розділі, і плутанина щодо неї — типова помилка новачків.

| Критерій | Неавтентифіковане | Автентифіковане |
|---|---|---|
| Точка зору | Як бачить систему анонімний зовнішній зловмисник | Як бачить систему користувач/адміністратор з обліковими даними |
| Що виявляє | Відкриті порти, банери служб, публічно доступні вразливості | Точні версії встановленого ПЗ, відсутні патчі ОС, слабкі локальні конфігурації, права доступу |
| Кількість хибних спрацювань (false positives) | Вища — сканер здогадується про версію ПЗ за непрямими ознаками | Нижча — сканер читає реальні дані пакетного менеджера/реєстру |
| Ризик для системи | Нижчий (менше активної взаємодії) | Потенційно вищий (потребує привілейованого облікового запису — сам по собі стає ціллю) |
| Типовий сценарій | Периметр, зовнішній vulnerability assessment | Внутрішня інфраструктура, підготовка до аудиту compliance |

> **Міні-вправа 12.3.1:** Організація проводить лише неавтентифіковане сканування зовнішнього периметра і на основі цього звітує керівництву «критичних вразливостей не виявлено». Які два конкретні ризики має ця практика?
>
> <details><summary>Відповідь</summary>
>
> 1. **Хибне відчуття безпеки щодо внутрішньої мережі** — неавтентифіковане зовнішнє сканування нічого не каже про стан робочих станцій, внутрішніх серверів, застарілих патчів на машинах, недоступних ззовні; саме звідти зазвичай починається lateral movement після фішингу (Модуль 07).
> 2. **Занижена точність навіть для периметра** — без автентифікації сканер часто не може підтвердити точну версію ПЗ (банер може бути змінений або прихований), тому реальні вразливості лишаються непоміченими, а не «відсутніми».
> </details>

## Глибина сканування та типові хибні спрацювання

Сканери працюють переважно за принципом **зіставлення сигнатур/версій** з базою CVE (через CPE-ідентифікатор виявленого ПЗ). Це породжує два типи помилок:

- **False Positive** — сканер повідомляє про вразливість, якої насправді немає (наприклад, версія ПЗ формально вразлива, але вендор бекпортував патч без зміни номера версії — типово для Red Hat Enterprise Linux).
- **False Negative** — сканер не повідомляє про реально наявну вразливість (наприклад, кастомний код застосунку, якого немає в жодній публічній базі CVE — сюди сканер не заглядає, на відміну від SAST/пентесту).

Тому результати сканера завжди потребують **тріажу (triage)**: аналітик перевіряє знайдені вразливості на релевантність перед передачею в чергу на усунення, інакше команди, відповідальні за патчинг, швидко втрачають довіру до звітів через «шум».

## Приклади інструментів і обґрунтування вибору

| Інструмент | Тип | Обґрунтування вибору |
|---|---|---|
| OpenVAS/GVM | Network, open-source | Безкоштовний, підходить для малих команд і навчання; менш точний за комерційні аналоги |
| Nessus (Tenable) | Network, комерційний | Галузевий стандарт, найбільша база плагінів, автентифіковане сканування «з коробки» |
| OWASP ZAP | Web application, open-source | Інтегрується в CI/CD (DAST gate), активно підтримується спільнотою OWASP |
| Trivy | Container/IaC, open-source | Швидкий, підтримує Docker-образи, Kubernetes-манифести та Terraform одночасно |
| Nikto | Web server, open-source | Легкий інструмент для швидкої перевірки конфігурації вебсервера (застарілі заголовки, небезпечні файли) |

## CIS-CAT та hardening-скани окремо від vulnerability-сканів

Важливо не плутати сканування вразливостей (CVE) із скануванням на відповідність (compliance/hardening scan), яке перевіряє конфігурацію системи проти еталонного бенчмарку, наприклад **CIS Benchmarks**. Перше запитує «чи є в цьому продукті відома дірка», друге — «чи налаштована ця система за best practices незалежно від наявності CVE» (наприклад, чи вимкнено анонімний FTP, чи встановлено мінімальну довжину пароля). Обидва підходи взаємодоповнюють одне одного і детальніше розглядаються в розділі 12.5.
