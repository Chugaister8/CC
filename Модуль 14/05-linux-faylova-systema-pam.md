# 14.5. Linux: файлова система, права доступу, PAM

## Перехід на паралельний Linux-трек

Розділи 14.3-14.4 розглянули Windows-специфічні механізми. Linux-серверна інфраструктура (переважна більшість хмарних робочих навантажень, вебсерверів, контейнерних платформ) вимагає паралельного, але концептуально відмінного набору практик hardening — почнемо з фундаменту: файлової системи і моделі прав доступу.

## Класична модель прав доступу Unix/Linux (DAC)

Кожен файл і директорія в Linux має власника (owner), групу (group) та права доступу для трьох категорій: власник, група, всі інші (others), кожна з трьох можливих прав: читання (r), запис (w), виконання (x). Це — приклад **Discretionary Access Control (DAC)**: власник ресурсу на власний розсуд (discretion) вирішує, кому надати доступ.

```
-rwxr-xr--  1 www-data  developers  /var/www/app/config.php
 |||||||||
 |||||||└─ others: read (r--)
 ||||└──── group: read+execute (r-x)
 └──────── owner: read+write+execute (rwx)
```

**Ключова слабкість DAC**, яку розділ 14.6 (SELinux/AppArmor) вирішує далі: якщо процес скомпрометовано (наприклад, вебсервер з правами `www-data`), він успадковує всі права цього користувача — DAC не має механізму обмежити, що саме скомпрометований процес може робити в межах прав власного облікового запису, навіть якщо ці дії явно не відповідають нормальній поведінці застосунку.

## Спеціальні біти прав: SUID, SGID, Sticky Bit

- **SUID (Set User ID)** — виконуваний файл із цим бітом запускається з правами **власника файлу**, а не користувача, що його запускає; класичний приклад — `/usr/bin/passwd`, що потребує прав root для зміни файлу паролів, навіть коли його запускає звичайний користувач для зміни власного пароля.
- **SGID (Set Group ID)** — аналогічно, але успадковує права групи-власника.
- **Sticky Bit** — застосовується до директорій (наприклад, `/tmp`), дозволяючи видаляти файли лише їх власнику, навіть якщо директорія доступна для запису всім.

**SUID-бінарники — критична поверхня атаки**: будь-який SUID-виконуваний файл, що належить root і містить вразливість (переповнення буфера, ін'єкцію команд, некоректну обробку аргументів), дає зловмиснику шлях до ескалації привілеїв від звичайного користувача до root. Регулярний аудит системи на предмет несанкціонованих чи зайвих SUID-бінарників (`find / -perm -4000 -type f`) — стандартна hardening-практика, прямо пов'язана з принципом мінімізації поверхні атаки.

> **Міні-вправа 14.5.1:** Аудит виявляє SUID-біт на нестандартному, самописному скрипті `/opt/legacy/backup.sh`, що належить root і був створений кілька років тому «для зручності», дозволяючи звичайним користувачам запускати резервне копіювання без sudo. Чому це серйозніша проблема, ніж SUID на стандартних системних утилітах на кшталт `passwd`?
>
> <details><summary>Відповідь</summary>
>
> Стандартні системні SUID-утиліти (passwd, sudo, ping) написані, ретельно перевірені й підтримуються великою спільнотою розробників дистрибутива, з довгою історією аудиту безпеки. Самописний shell-скрипт із SUID-бітом, що виконується з правами root, майже завжди становить набагато вищий ризик: shell-скрипти особливо вразливі до ін'єкції команд, race conditions і небезпечної обробки змінних середовища, а відсутність формального аудиту безпеки означає, що будь-яка неочевидна вада в логіці скрипта — це прямий шлях до root-доступу для будь-якого користувача, здатного його запустити. Найкраща практика — ніколи не встановлювати SUID на власні скрипти; замість цього використовувати `sudo` з детально обмеженим `sudoers`-правилом (нижче), що дає аудитований, контрольований шлях підвищення привілеїв лише для конкретної команди.
> </details>

## sudo та принцип найменших привілеїв

`sudo` (superuser do) — контрольований, аудитований механізм тимчасового підвищення привілеїв, що є кращою альтернативою SUID-скриптам чи постійним root-сесіям, реалізуючи принцип найменших привілеїв (least privilege, уже згадуваний у контексті архітектурних принципів секцій цього посібника):

```
# /etc/sudoers (редагується виключно через visudo)
# Дозволяє групі 'backup-operators' виконувати ЛИШЕ конкретний скрипт
# резервного копіювання від імені root, без пароля, з повним журналюванням
%backup-operators ALL=(root) NOPASSWD: /opt/scripts/run-backup.sh
```

Hardening-практики для sudo:

- **Уникати `ALL=(ALL) ALL`** — надання необмежених прав sudo без конкретизації команд еквівалентно постійному root-доступу з додатковим кроком уведення пароля.
- **Журналювання через `sudo` завжди активне** (`/var/log/auth.log` чи через syslog) — кожен виклик sudo фіксується з часовою міткою, користувачем і точною командою, що є критичним джерелом даних для журналювання (розділ 14.9).
- **`Defaults use_pty`** — примушує sudo виконувати команди в псевдотерміналі, ускладнюючи спроби обходу журналювання через певні техніки маніпуляції процесами.

## PAM (Pluggable Authentication Modules)

**PAM** — модульна архітектура автентифікації в Linux, що дозволяє гнучко налаштовувати механізм перевірки особи (пароль, біометрія, апаратний токен, MFA) незалежно від конкретного застосунку (SSH, login, sudo всі використовують той самий централізований стек PAM-модулів). Ключові hardening-конфігурації PAM:

- **`pam_faillock` / `pam_tally2`** — блокування облікового запису після визначеної кількості невдалих спроб входу (аналог Account Lockout Policy з Windows Group Policy, розділ 14.3).
- **`pam_pwquality`** — вимоги до складності пароля (мінімальна довжина, наявність цифр/спецсимволів) — конфігураційний аналог Windows Password Policy.
- **`pam_google_authenticator` чи `pam_oath`** — інтеграція TOTP-based MFA (пряме продовження теми MFA з Модуля 05) безпосередньо на рівні автентифікації SSH чи локального входу, без потреби у зовнішньому IdP.

## SSH hardening: найважливіший вхідний вектор Linux-сервера

SSH — типово основний, а часто єдиний віддалений вхід до Linux-сервера, що робить його hardening пріоритетним:

- **Вимкнення автентифікації по паролю, лише SSH-ключі** (`PasswordAuthentication no`) — усуває цілий клас brute-force атак.
- **Вимкнення прямого входу root** (`PermitRootLogin no`) — вимагає входу під звичайним користувачем з подальшим `sudo`, забезпечуючи атрибуцію дій до конкретної особи (пряма підтримка журналювання й розслідування інцидентів).
- **Зміна стандартного порту 22** — не є справжнім захисним контролем (security through obscurity), лише знижує шум від автоматизованого масового сканування; не замінює жодного з двох попередніх пунктів.
- **`fail2ban`** — інструмент, що автоматично блокує IP-адреси після повторюваних невдалих спроб автентифікації, аналогічний Account Lockout Policy, але на мережевому рівні.
