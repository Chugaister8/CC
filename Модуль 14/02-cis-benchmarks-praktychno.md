# 14.2. CIS Benchmarks практично

## Від згадки в Модулі 12 до практичного застосування

Модуль 12 (розділ 12.5) вперше згадав CIS Benchmarks як інструмент, що відповідає на питання «чи налаштована система за best practices незалежно від наявності CVE». Цей розділ розкриває, як саме структуровані ці бенчмарки і як їх застосовують практично — методологічна основа для всіх наступних розділів модуля.

## Структура CIS Benchmarks

**Center for Internet Security (CIS)** публікує безкоштовні, детальні бенчмарки для десятків платформ (Windows Server, Windows десктопні редакції, основні Linux-дистрибутиви, хмарні платформи AWS/Azure/GCP, бази даних, контейнерні платформи). Кожен бенчмарк — документ на сотні сторінок з сотнями окремих, пронумерованих пунктів-рекомендацій, кожен з яких містить:

- **Опис (Description)** — що саме перевіряється чи налаштовується.
- **Обґрунтування (Rationale)** — чому ця настройка важлива з точки зору безпеки.
- **Аудит (Audit)** — конкретна команда чи крок для перевірки поточного стану.
- **Виправлення (Remediation)** — конкретна команда чи крок для приведення до безпечного стану.
- **Вплив на дозвільність (Impact)** — можливі побічні ефекти зміни (важливо: деякі рекомендації можуть порушити сумісність із застарілим ПЗ).

## Рівні (Profiles): Level 1 проти Level 2

Кожен бенчмарк ділить рекомендації на два профілі:

- **Level 1 (Базовий)** — рекомендації, що суттєво знижують поверхню атаки з **мінімальним ризиком порушення функціональності** чи сумісності; призначені для застосування практично в будь-якому середовищі без суттєвого додаткового тестування.
- **Level 2 (Поглиблений)** — рекомендації для середовищ з підвищеними вимогами до безпеки (оборонна промисловість, фінансовий сектор, критична інфраструктура), що можуть обмежити певну функціональність чи вимагати додаткової сумісності з іншим ПЗ; вимагають ретельнішого тестування перед впровадженням у продакшн.

**Практичне правило:** Level 1 — типовий обов'язковий мінімум практично для будь-якої організації; Level 2 застосовується вибірково, з оцінкою компромісу між безпекою й операційною зручністю (пряме продовження принципу пропорційності з блоку контексту завдання).

> **Міні-вправа 14.2.1:** Бенчмарк CIS для Windows Server рекомендує на рівні Level 2 повне вимкнення протоколу SMBv1 навіть для внутрішньої мережі. Компанія має один застарілий виробничий верстат із вбудованим ПЗ 2008 року, що працює виключно через SMBv1 і не підлягає оновленню (виробник збанкрутував). Яке рішення відповідає принципу пропорційності з цього посібника, і як це задокументувати відповідно до практик Модуля 13?
>
> <details><summary>Відповідь</summary>
>
> Пряме, сліпе застосування рекомендації Level 2 у цьому випадку зупинило б критичний виробничий процес — неприйнятний компроміс. Пропорційне рішення: вимкнути SMBv1 всюди, **крім** сегменту мережі, ізольованого для цього конкретного застарілого верстата (мережева сегментація як компенсуючий контроль, аналогічно віртуальному патчуванню з Модуля 12, розділ 12.4), і задокументувати це відхилення від бенчмарку як формально прийнятий залишковий ризик у реєстрі ризиків (Модуль 13, розділ 13.7-13.8) з чітким обґрунтуванням і власником рішення, а не залишати непояснене відхилення непомітним.
> </details>

## Інструменти автоматизованої перевірки

Ручна перевірка сотень пунктів бенчмарку для кожного сервера нежиттєздатна на масштабі — застосовуються спеціалізовані інструменти:

- **CIS-CAT (Configuration Assessment Tool)** — офіційний інструмент CIS, що автоматично сканує систему проти конкретного бенчмарку й видає звіт відповідності у вигляді **CIS Benchmark Score** (відсоток пунктів, що відповідають рекомендаціям).
- **OpenSCAP** — відкритий інструмент реалізації стандарту **SCAP (Security Content Automation Protocol)**, що дозволяє автоматизовану перевірку відповідності конфігурації на Linux-системах за профілями, включно з CIS та іншими стандартами (наприклад, DISA STIG — американський військовий еквівалент CIS для оборонного сектору).
- **Microsoft Security Compliance Toolkit** — офіційні базові конфігурації (security baselines) від Microsoft у форматі Group Policy Objects (GPO), які частково перетинаються з рекомендаціями CIS для Windows і можуть імпортуватися напряму (детально — розділ 14.3).

## CIS Controls проти CIS Benchmarks: не плутати

Важлива термінологічна відмінність, яку варто чітко розрізняти:

| | CIS Controls | CIS Benchmarks |
|---|---|---|
| Рівень абстракції | Високорівневі організаційні практики (18 контролів, наприклад Control 1 «Inventory and Control of Enterprise Assets», уже згадуваний у Модулях 12-13) | Конкретні, деталізовані технічні налаштування для конкретного продукту/ОС |
| Приклад | «Організація повинна вести реєстр активів» | «Встановити параметр `MinimumPasswordLength` не менше 14 символів у Windows Server 2022» |
| Аналогія | Стратегія | Тактика реалізації |

CIS Benchmarks — технічна, платформо-специфічна реалізація ширших організаційних принципів CIS Controls; цей модуль зосереджений саме на Benchmarks-рівні деталізації.

## Практичний робочий процес впровадження

1. Обрати відповідний бенчмарк і профіль (Level 1/2) на основі класифікації активу (Модуль 13, розділ 13.3).
2. Провести базове сканування (CIS-CAT/OpenSCAP) для отримання поточного стану відповідності.
3. Пріоритизувати відхилення за впливом на безпеку й ризиком порушення функціональності (аналогічно пріоритизації вразливостей у Модулі 12, розділ 12.4).
4. Тестувати зміни в staging-середовищі перед продакшн-розгортанням (той самий принцип, що й для патчів, Модуль 12, розділ 12.5).
5. Автоматизувати підтримку відповідності через Infrastructure as Code (розділ 14.7), а не разове ручне налаштування, яке з часом «дрейфує» від безпечного стану.
