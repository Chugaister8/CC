# 14.6. Linux: SELinux та AppArmor

## Вирішення слабкості, окресленої в попередньому розділі

Розділ 14.5 завершився ключовим обмеженням DAC: якщо процес скомпрометовано, він успадковує всі права свого користувача без додаткових обмежень щодо *очікуваної* поведінки. **Mandatory Access Control (MAC)** — принципово інша модель, що вирішує саме цю проблему: система (а не власник ресурсу) примусово визначає, що саме конкретному процесу дозволено робити, незалежно від прав користувача, від імені якого цей процес запущено.

## DAC проти MAC: концептуальна різниця

| | DAC (стандартні Unix-права, розділ 14.5) | MAC (SELinux/AppArmor) |
|---|---|---|
| Хто визначає права | Власник ресурсу, на власний розсуд | Централізована системна політика, обов'язкова для всіх |
| Одиниця контролю | Користувач/група | Процес (точніше — контекст/домен, до якого прив'язаний процес) |
| Чи може користувач змінити правила для власних файлів | Так (chmod) | Ні — політика встановлюється адміністратором системи централізовано |
| Захист від скомпрометованого процесу з легітимними правами користувача | Немає — процес успадковує всі права користувача | Є — процес обмежений політикою навіть у межах прав свого користувача |

**Практичний приклад:** вебсервер Apache, що працює під користувачем `www-data`, технічно має права DAC читати будь-який файл, доступний для читання групі чи всім (наприклад, `/etc/shadow`, якщо неправильно налаштовано, чи файли інших вебзастосунків на тому самому сервері). SELinux-політика може обмежити процес Apache так, щоб він **фізично не міг** читати файли поза власною, явно дозволеною директорією `/var/www/html`, незалежно від того, які DAC-права формально дозволяють — навіть якщо вебзастосунок скомпрометовано через вразливість (наприклад, LFI з Модуля 06), зловмисник не зможе вийти за межі дозволеного SELinux-домену.

## SELinux (Security-Enhanced Linux)

Розроблений спочатку NSA, тепер стандартна частина Red Hat Enterprise Linux, CentOS, Fedora та похідних дистрибутивів. Ключові концепції:

- **Контексти безпеки (Security Contexts)** — кожен файл, процес, порт має мітку у форматі `user:role:type:level` (наприклад, `system_u:object_r:httpd_sys_content_t:s0` для файлів вебконтенту). Найважливіший компонент для практичного адміністрування — **type** (тип), що визначає, до яких дій щодо цього об'єкта застосовується політика.
- **Домени (Domains)** — процеси працюють у певному домені (типі), і політика визначає, які домени можуть взаємодіяти з якими типами файлів/портів/інших процесів.
- **Режими роботи**:
  - **Enforcing** — політика активно застосовується, порушення блокуються й журналюються.
  - **Permissive** — порушення лише журналюються, але не блокуються (корисно для тестування нової політики без ризику зламати функціональність).
  - **Disabled** — SELinux повністю вимкнено (типова, але небезпечна практика адміністраторів, що стикаються з незрозумілими помилками «Permission denied» без діагностики).

**Типова помилка адміністраторів-новачків:** зіткнувшись із незрозумілою відмовою в доступі, викликаною SELinux, багато хто вимикає SELinux повністю (`setenforce 0` назавжди) замість діагностики й виправлення конкретної політики — це усуває цілий рівень захисту через незручність, а не через обґрунтоване рішення про ризик. Правильний підхід: `audit2allow` — утиліта, що аналізує журнал відмов SELinux (`/var/log/audit/audit.log`, розділ 14.9) і генерує **точний, мінімально необхідний** політичний модуль-виняток для конкретної легітимної дії, що була заблокована помилково, зберігаючи захист для всього іншого.

> **Міні-вправа 14.6.1:** Новий вебзастосунок після розгортання показує помилку `Permission denied` при спробі підключення до нестандартного порту бази даних (не типового 3306/5432). Journalctl показує AVC (Access Vector Cache) denial від SELinux. Який із двох підходів — `setenforce 0` чи `audit2allow` з подальшим `semanage port` для дозволу конкретного порту — відповідає принципу мінімізації поверхні атаки, і чому?
>
> <details><summary>Відповідь</summary>
>
> Правильний підхід — діагностика через AVC-журнал і застосування точного, мінімального виправлення (`semanage port -a -t mysqld_port_t -p tcp <НОМЕР_ПОРТУ>` для дозволу конкретного нестандартного порту в межах відповідного домену), а не `setenforce 0`. Перший підхід зберігає весь захист SELinux для решти системи, дозволяючи лише одну конкретну, усвідомлену дію; другий повністю вимикає Mandatory Access Control для абсолютно всіх процесів системи заради вирішення однієї вузької проблеми — це пряме порушення принципу мінімізації поверхні атаки заради операційної зручності, аналогічне сліпому вимкненню файрвола через незрозумілу помилку з'єднання замість діагностики конкретного правила.
> </details>

## AppArmor: простіша альтернатива

**AppArmor** (стандартний для Ubuntu, Debian, SUSE) реалізує ту саму ідею MAC, але з простішою, більш доступною моделлю на основі **шляхів файлової системи** замість абстрактних міток типів SELinux:

```
# /etc/apparmor.d/usr.sbin.nginx (спрощений приклад профілю)
/usr/sbin/nginx {
  /etc/nginx/** r,
  /var/www/html/** r,
  /var/log/nginx/*.log w,
  network inet stream,
  deny /etc/shadow r,
  deny /home/** rwx,
}
```

Профіль AppArmor явно перелічує, до яких шляхів процес має доступ і з якими правами, за замовчуванням забороняючи все, що не перелічено — той самий принцип secure defaults / allowlisting, що й у Windows AppLocker (розділ 14.3).

**Компроміс простоти й гнучкості:** AppArmor легше вивчити й адмініструвати (профілі читабельні, прив'язані до знайомих шляхів файлової системи), але SELinux дає точніший, детальніший контроль (мітки типів дозволяють значно гнучкішу, менш прив'язану до конкретних шляхів політику) — типовий компроміс між зручністю використання й глибиною контролю, знайомий з інших областей безпеки в цьому посібнику.

## Практичне застосування: контейнери

MAC-механізми критично важливі не лише для традиційних серверів, а й для ізоляції контейнерів (Docker, Kubernetes): попри те, що контейнери створюють ілюзію ізольованого середовища через простори імен ядра (namespaces) та control groups (cgroups), сам механізм ізоляції контейнерів **не є** MAC-системою — Docker і Kubernetes на більшості дистрибутивів додатково покладаються саме на SELinux чи AppArmor профілі як додатковий шар захисту від container escape (виходу процесу за межі контейнера на хостову систему) — практичний місток до безпеки контейнерів, коротко згаданої в Модулі 06.
