# Модуль 14. Безпека операційних систем (Windows і Linux hardening)

> Модуль 13 навчив вас, як приймати обґрунтовані рішення про те, куди інвестувати обмежені ресурси безпеки. Модуль 12 (розділ 12.5) побіжно згадав CIS Benchmarks і hardening як практику, що доповнює патчинг. Цей модуль розкриває цю практику повністю: операційна система — фундамент, на якому стоїть кожен контроль з попередніх модулів (криптографія, автентифікація, вебзастосунки, мережа). Найдосконаліша криптографічна схема (Модуль 04) марна, якщо зловмисник отримує root-доступ через незахищений SSH; найретельніша модель RBAC (Модуль 05) обходиться локальною ескалацією привілеїв через невиправлену конфігурацію Windows. Hardening ОС — не окрема галузь, а той шар, без якого всі інші шари захисту стоять на піску.

## Цілі модуля

Після опрацювання модуля ви зможете:

- **Запам'ятати:** структуру моделі безпеки Windows (токени доступу, ACL, integrity levels); принципи роботи SELinux/AppArmor; рівні CIS Benchmarks (Level 1/Level 2); компоненти tiered-моделі адміністрування.
- **Зрозуміти:** пояснити різницю між DAC та MAC на прикладі Linux; описати, чому Group Policy сама по собі не гарантує застосування налаштувань без моніторингу відхилень (configuration drift); пояснити, як EDR технічно відрізняється від класичного антивірусу на рівні ядра ОС.
- **Застосувати:** провести базовий hardening-аудит Linux-системи за CIS Benchmark вручну та через скрипт; налаштувати базові правила аудиту Linux (auditd) і проаналізувати Windows Event Log/Sysmon-подібні дані; написати Python-скрипт для перевірки відповідності системи базовим критеріям hardening.

**Орієнтовна тривалість:** 14–16 годин.

## Передумови

Рекомендовано пройти Модуль 05 (автентифікація, Kerberos, RBAC), Модуль 08 (SELinux/AppArmor у мобільному контексті, MDM), Модуль 12 (patch management, CIS Benchmarks — базове знайомство) та Модуль 13 (управління ризиками — для розуміння, чому hardening пріоритизується саме так).

## Практичне застереження

Приклади конфігурацій цього модуля — навчальні відправні точки, а не готові продакшн-конфігурації «під копірку». Кожне hardening-налаштування має бути протестоване в staging-середовищі (Модуль 12, розділ 12.5) перед застосуванням до продакшн-систем — надмірно агресивний hardening без тестування регулярно призводить до відмови легітимних бізнес-процесів.
