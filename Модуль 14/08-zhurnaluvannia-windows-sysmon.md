# 14.8. Журналювання Windows: Event Log та Sysmon

## Від профілактики до видимості

Розділи 14.2-14.7 зосереджувалися на профілактиці: як налаштувати систему так, щоб атака не вдалася. Але жоден набір превентивних контролів не гарантує стовідсоткового захисту (принцип залишкового ризику з Модуля 13). Журналювання відповідає на наступне логічне запитання: **якщо профілактика все ж не спрацювала, чи побачимо ми це взагалі, і чи достатньо швидко?** Це та сама логіка, що обґрунтовувала Purple Team у Модулі 12 (розділ 12.9) — детекція так само важлива, як профілактика.

## Windows Event Log: вбудований, але обмежений базовий рівень

Стандартний **Windows Event Log** (переглядається через Event Viewer чи PowerShell `Get-WinEvent`) фіксує події за замовчуванням у кількох журналах: Security (входи в систему, зміни політик), System (події ОС), Application (події застосунків). Ключові Event ID для безпекового моніторингу:

| Event ID | Опис |
|---|---|
| 4624 | Успішний вхід у систему |
| 4625 | Невдала спроба входу (потенційний brute-force) |
| 4672 | Вхід з привілейованими правами (адміністративний обліковий запис) |
| 4720 | Створення нового облікового запису користувача |
| 4732 | Додавання користувача до привілейованої локальної групи |
| 4688 | Створення нового процесу |

**Фундаментальне обмеження стандартного Event Log:** за замовчуванням він фіксує **факт** події (хтось увійшов, процес створено), але вкрай обмежено фіксує **контекст** цієї події — наприклад, Event ID 4688 (створення процесу) стандартно не включає повний рядок команди (command line), з яким процес був запущений, що робить неможливим розрізнення легітимного й підозрілого виклику одного й того самого системного інструменту (наприклад, PowerShell, запущений із нормальним скриптом, проти PowerShell із обфускованим, шкідливим одноlainer-ом — типова техніка LOLBAS з Модуля 07).

## Sysmon: заповнення прогалини деталізації

**Sysmon (System Monitor)**, частина набору **Sysinternals** від Microsoft, — не заміна Event Log, а доповнення, що фіксує значно детальнішу телеметрію, спеціально призначену для виявлення підозрілої активності:

- **Event ID 1** — створення процесу, **з повним рядком команди, хешем виконуваного файлу та ім'ям батьківського процесу** (вирішує обмеження Event ID 4688 вище).
- **Event ID 3** — мережеве з'єднання, ініційоване процесом (критично для виявлення command-and-control з'єднань, Модуль 07).
- **Event ID 7** — завантаження образу/DLL — виявлення DLL side-loading та ін'єкції коду.
- **Event ID 8** — CreateRemoteThread — класична техніка process injection, часто використовувана для приховування шкідливого коду всередині легітимного процесу.
- **Event ID 11** — створення файлу — корисно для виявлення дропу шкідливих файлів у нетипових локаціях.
- **Event ID 22** — DNS-запит — виявлення підозрілих доменів (наприклад, DGA — Domain Generation Algorithm, техніка, що використовується деякими родинами malware для command-and-control).

## Конфігурація Sysmon: XML-файл правил

Sysmon без конфігурації генерує величезний обсяг шуму (кожне мережеве з'єднання й запуск процесу). Практичне налаштування використовує конфігураційний XML-файл із правилами include/exclude:

```xml
<!-- Спрощений приклад Sysmon-конфігурації:
     виявлення підозрілого запуску PowerShell з обфускацією -->
<Sysmon schemaversion="4.90">
  <EventFiltering>
    <ProcessCreate onmatch="include">
      <CommandLine condition="contains">-EncodedCommand</CommandLine>
      <CommandLine condition="contains">-nop -w hidden</CommandLine>
      <CommandLine condition="contains">DownloadString</CommandLine>
    </ProcessCreate>
    <NetworkConnect onmatch="include">
      <Image condition="end with">powershell.exe</Image>
      <Image condition="end with">certutil.exe</Image>
    </NetworkConnect>
  </EventFiltering>
</Sysmon>
```

Спільнота підтримує готові, широко перевірені конфігурації (найвідоміша — **SwiftOnSecurity Sysmon config**), що охоплюють десятки відомих технік MITRE ATT&CK (Модуль 07/12) без необхідності писати правила з нуля.

> **Міні-вправа 14.8.1:** Аналітик SOC бачить у стандартному Windows Event Log (Event ID 4688) лише запис «створено процес powershell.exe», без додаткового контексту. У паралельному потоці Sysmon Event ID 1 показує повний рядок команди: `powershell.exe -nop -w hidden -EncodedCommand SQBFAFgAKABOAGUAdw...`. Чому саме друге джерело дозволяє негайно кваліфікувати цю подію як підозрілу, тоді як перше — ні?
>
> <details><summary>Відповідь</summary>
>
> Стандартний Event Log фіксує лише факт запуску `powershell.exe` — легітимна, надзвичайно поширена дія, що сама по собі нічого не говорить про намір. Sysmon-запис показує конкретні прапори командного рядка: `-nop` (без профілю), `-w hidden` (прихована консоль), `-EncodedCommand` (Base64-закодована команда, класична техніка обфускації, щоб приховати реальний вміст скрипта від простого візуального огляду й від сигнатурного антивірусного сканування) — комбінація цих трьох прапорів є настільки типовим патерном шкідливого використання PowerShell (частина LOLBAS-техніки з Модуля 07), що більшість SOC-плейбуків автоматично ескалують такі події для негайного розслідування, тоді як стандартний Event Log фізично не містить даних, необхідних для такого висновку.
> </details>

## Windows Event Forwarding (WEF) та централізація

Локальні журнали на кожній машині марні для системного моніторингу без централізації — **Windows Event Forwarding** дозволяє автоматично пересилати обрані події (включно з Sysmon-подіями) з тисяч кінцевих точок на центральний collector-сервер, звідки вони потрапляють до SIEM-системи (Security Information and Event Management) для кореляції, алертингу й довгострокового зберігання — необхідна умова для практик Purple Team і BAS, розглянутих у Модулі 12 (розділ 12.9): без централізованого журналювання неможливо перевірити, чи справді детектується конкретна симульована атака.
