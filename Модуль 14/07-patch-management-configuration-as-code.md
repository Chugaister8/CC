# 14.7. Патч-менеджмент та конфігурація як код

## Сходження двох треків на спільній темі

Розділи 14.3-14.6 розгалужувалися на паралельні Windows та Linux треки. Цей розділ — перша точка сходження: патч-менеджмент і автоматизація конфігурації — теми, що застосовуються однаково концептуально до обох платформ, хоча й з різними інструментами.

## Патч-менеджмент на рівні ОС: специфіка масштабу

Модуль 12 (розділ 12.5) уже детально розглянув загальний процес Patch Management: SLA, staging-тестування, вікна обслуговування. Цей підрозділ додає специфіку саме для операційних систем на масштабі парку з сотень чи тисяч машин.

### Windows: WSUS та Windows Update for Business

- **WSUS (Windows Server Update Services)** — локальний сервер, що завантажує оновлення від Microsoft один раз, дозволяючи адміністратору схвалювати конкретні патчі для конкретних груп комп'ютерів перед розповсюдженням, замість того, щоб кожна машина завантажувала оновлення напряму з інтернету — центральний контроль над тим, що й коли встановлюється.
- **Windows Update for Business / Intune** (уже згадувані в Модулі 08 у контексті MDM) — хмарна альтернатива WSUS для сучасних, керованих через хмару парків пристроїв, з кільцями розгортання (deployment rings): невелика група пілотних машин отримує патч першою, і лише після підтвердження відсутності проблем патч поширюється на всю організацію.

### Linux: unattended-upgrades та управління пакетами

- **`unattended-upgrades`** (Debian/Ubuntu) чи `dnf-automatic` (RHEL/Fedora) — автоматичне встановлення оновлень безпеки без ручного втручання, з можливістю обмежити автоматизм лише критичними патчами безпеки, залишаючи функціональні оновлення для планового ручного циклу тестування.
- **Kernel Live Patching** (`kpatch` для RHEL, `livepatch` для Ubuntu) — унікальна для Linux можливість застосувати патч безпеки безпосередньо до працюючого ядра **без перезавантаження системи**, критично важлива для серверів з вимогами максимальної доступності (high-availability систем, де навіть коротке планове перезавантаження для оновлення ядра неприйнятне з точки зору RTO, Модуль 13, розділ 13.9).

> **Міні-вправа 14.7.1:** Критичний сервер бази даних має задокументований RTO = 5 хвилин (Модуль 13), але звичайний процес патчингу ядра вимагає перезавантаження, що займає 15-20 хвилин. Яке технічне рішення з цього розділу вирішує це протиріччя, і чому воно не є універсальною заміною традиційного патчингу назавжди?
>
> <details><summary>Відповідь</summary>
>
> Kernel Live Patching дозволяє застосувати критичний патч безпеки ядра без перезавантаження, укладаючись у 5-хвилинний RTO, оскільки сервіс залишається доступним протягом усього процесу патчингу. Проте це не універсальна заміна: Live Patching зазвичай підтримує лише підмножину типів патчів (переважно виправлення безпеки, що не змінюють фундаментальну структуру даних ядра); складніші оновлення все ще вимагають традиційного перезавантаження при плановому вікні обслуговування (Модуль 12, розділ 12.5) — Live Patching купує час до наступного планового вікна, а не усуває потребу в ньому назавжди.
> </details>

## Configuration as Code: від разового налаштування до відтворюваного стану

Розділ 14.3 коротко згадав PowerShell DSC як доповнення до Group Policy. Ширша концепція — **Infrastructure as Code (IaC)** для конфігурації безпеки, що вирішує фундаментальну проблему ручного hardening: **configuration drift** (дрейф конфігурації) — поступове, непомітне розходження між задокументованим безпечним станом системи й реальним поточним станом, що накопичується через ручні зміни адміністраторів, тимчасові виправлення «на швидку руку», що ніколи не відкочуються, і людські помилки.

### Основні інструменти

| Інструмент | Платформа | Підхід |
|---|---|---|
| **Ansible** | Крос-платформний (Windows/Linux) | Декларативні YAML-плейбуки, безагентна архітектура (через SSH/WinRM) |
| **Chef** | Крос-платформний | Декларативні «рецепти» на Ruby-подібній DSL, агентна архітектура |
| **Puppet** | Крос-платформний | Декларативна модель, агентна архітектура з періодичною перевіркою відповідності |
| **PowerShell DSC** | Windows-специфічний | Декларативні конфігурації, глибока інтеграція з Group Policy |

**Принцип роботи всіх цих інструментів однаковий:** адміністратор описує **бажаний кінцевий стан** системи (наприклад, «SSH-сервіс має мати `PasswordAuthentication no`, порт 22 закритий файрволом ззовні, пакет `telnet` видалений») декларативно, а не покроково («виконати цю команду, потім оцю»). Інструмент періодично перевіряє реальний стан кожної машини проти цього опису й автоматично виправляє будь-яке відхилення — так конфігураційний дрейф виявляється й усувається автоматично, а не через рідкісний ручний аудит.

```yaml
# Спрощений приклад Ansible-плейбуку для базового SSH hardening
# (розділ 14.5) - декларативний опис бажаного стану, не покрокові команди
- name: SSH hardening baseline
  hosts: all_linux_servers
  tasks:
    - name: Disable password authentication
      lineinfile:
        path: /etc/ssh/sshd_config
        regexp: '^#?PasswordAuthentication'
        line: 'PasswordAuthentication no'
      notify: restart sshd

    - name: Disable direct root login
      lineinfile:
        path: /etc/ssh/sshd_config
        regexp: '^#?PermitRootLogin'
        line: 'PermitRootLogin no'
      notify: restart sshd

  handlers:
    - name: restart sshd
      service:
        name: sshd
        state: restarted
```

## Інтеграція з CIS Benchmarks: готові плейбуки відповідності

Замість написання hardening-конфігурації з нуля, спільнота й комерційні постачальники пропонують готові, підтримувані Ansible-ролі та DSC-конфігурації, що прямо реалізують конкретні CIS Benchmark рівні (наприклад, публічно доступні ролі `ansible-lockdown` для CIS-відповідності RHEL/Ubuntu) — практичний місток між методологією розділу 14.2 (що саме перевіряти) і автоматизованою, відтворюваною реалізацією цього розділу (як гарантовано підтримувати цей стан на масштабі без ручної праці й дрейфу).

## Зв'язок із наступними розділами

Автоматизована, відтворювана конфігурація вирішує проблему підтримки безпечного стану, але не вирішує іншу критичну проблему: **як дізнатися, коли хтось намагається порушити цей стан чи скористатися вразливістю, попри всі превентивні заходи**. Це — тема журналювання ОС, розглянута в наступних двох розділах.
