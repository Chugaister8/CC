# 14.13. Чек-лист і самоперевірка

## Чек-лист hardening операційних систем

### Методологія та базовий профіль

- [ ] Обрано конкретний CIS Benchmark (Windows/Linux, відповідна версія) як базовий профіль
- [ ] Чітко розділено застосування Level 1 (обов'язковий мінімум) і Level 2 (вибіркове посилення з оцінкою компромісу)
- [ ] Кожна hardening-зміна протестована в staging-середовищі перед продакшн-розгортанням (Модуль 12, розділ 12.5)

### Windows

- [ ] Group Policy (GPO) використовується для централізованого, автоматично повторюваного застосування налаштувань безпеки
- [ ] RSOP регулярно перевіряється для виявлення конфліктів чи неочікуваного результату застосування кількох GPO
- [ ] LSA Protection (PPL) увімкнена для базового захисту LSASS від дампінгу облікових даних
- [ ] Credential Guard увімкнена на системах, де вимоги сумісності це дозволяють
- [ ] Облікові записи найвищого рівня привілеїв входять до групи Protected Users
- [ ] LAPS впроваджена для ротації локальних адміністративних паролів

### Linux

- [ ] SSH сконфігурований без прямого root-входу та без парольної автентифікації (лише ключі)
- [ ] Регулярний аудит SUID/SGID-бінарників на предмет несанкціонованих чи зайвих записів
- [ ] Відсутні world-writable файли в критичних системних директоріях
- [ ] sudo налаштований за принципом найменших привілеїв, без надлишкових `NOPASSWD` записів
- [ ] PAM-стек сконфігурований з політикою складності паролів і блокування після невдалих спроб
- [ ] SELinux чи AppArmor активний у режимі enforcing/complain, а не вимкнений через незручність

### Патч-менеджмент та конфігурація

- [ ] Централізований інструмент патч-менеджменту (WSUS/Windows Update for Business для Windows, unattended-upgrades чи еквівалент для Linux) впроваджено
- [ ] Configuration as Code (Ansible/Puppet/Chef/DSC) використовується для виявлення та автоматичного виправлення дрейфу конфігурації

### Журналювання

- [ ] Sysmon розгорнутий на Windows-системах з конфігурацією, орієнтованою на конкретні безпекові сценарії, а не безрозбірливе журналювання всього
- [ ] auditd налаштований з цілеспрямованими правилами для критичних шляхів (`/etc/passwd`, `/etc/sudoers`, завантаження модулів ядра)
- [ ] Журнали централізовано пересилаються в SIEM (через WEF для Windows, rsyslog/форвардинг для Linux)

### EDR та привілейований доступ

- [ ] EDR-агент розгорнутий на всіх кінцевих точках з підпискою на ETW (Windows) чи еквівалентну телеметрію ядра (eBPF на Linux)
- [ ] Впроваджена Tiered Administration Model з чітким розділенням Tier 0/1/2
- [ ] Privileged Access Workstations використовуються для адміністративних завдань найвищого рівня
- [ ] Привілейований доступ надається через Just-In-Time модель, а не постійні привілеї

## Питання для самоперевірки

**1. Чому запатчена система (нуль відкритих CVE) не є тим самим, що захищена система?**

<details><summary>Відповідь</summary>

Патчинг закриває конкретні, каталогізовані технічні вразливості (Модуль 12). Hardening приводить базову конфігурацію системи до безпечного стану незалежно від наявності CVE — вимкнені непотрібні служби, обмежені привілеї, увімкнене журналювання. Система може не мати жодної невиправленої CVE, але залишатися вразливою через дозвільну конфігурацію за замовчуванням (надлишкові привілеї, відсутнє журналювання, незахищений LSASS) — саме цю прогалину демонструє кейс NotPetya (розділ 14.1), де вхідний вектор було закрито патчем, але подальше поширення відбулося через незахищені облікові дані в пам'яті.
</details>

**2. У чому принципова відмінність DAC від MAC, і чому наявність DAC не робить MAC зайвим?**

<details><summary>Відповідь</summary>

DAC — власник ресурсу на власний розсуд вирішує, кому надати доступ; одиниця контролю — користувач/група. MAC — централізована системна політика примусово визначає дозволену поведінку процесу незалежно від DAC-прав користувача, від імені якого він запущений. MAC не зайвий, тому що DAC має фундаментальну слабкість: якщо процес скомпрометовано, він успадковує всі DAC-права свого користувача без додаткових обмежень щодо очікуваної поведінки. MAC (SELinux/AppArmor) обмежує скомпрометований процес навіть у межах DAC-дозволених прав — два шари захисту, а не альтернативи один одному.
</details>

**3. Чому Credential Guard технічно ефективніший за просту LSA Protection (PPL) проти дампінгу облікових даних?**

<details><summary>Відповідь</summary>

LSA Protection (PPL) обмежує, які процеси можуть відкрити дескриптор доступу до пам'яті LSASS, але секрети (NTLM-хеші, Kerberos-квитки) фізично залишаються в пам'яті звичайного LSASS-процесу — просунуті техніки (наприклад, BYOVD, розділ 14.4) можуть обійти цей бар'єр, отримавши доступ на рівні ядра. Credential Guard використовує апаратну віртуалізацію (VBS), щоб фізично перемістити найчутливіші секрети в окремий, ізольований віртуалізований контейнер, до якого навіть процес із правами SYSTEM на основній ОС не має прямого доступу — LSASS на основній ОС більше не зберігає ці секрети напряму.
</details>

**4. Чому широкі, невибіркові правила auditd чи неналаштований Sysmon вважаються проблемою, а не просто «більше безпеки»?**

<details><summary>Відповідь</summary>

Надлишковий, невибірковий обсяг журналювання (наприклад, кожен системний виклик `open()` на всій файловій системі) створює настільки великий обсяг даних, що аналітик чи автоматизоване правило кореляції SIEM фізично не може виділити дійсно значущі події серед шуму — ефект, аналогічний хибним спрацюванням сканерів вразливостей (Модуль 12, розділ 12.3), що підривають довіру до системи. Ефективна конфігурація цілеспрямовано журналює конкретні, безпеково значущі шляхи й дії, а не намагається журналювати абсолютно все — принцип, спільний і для Sysmon, і для auditd.
</details>

**5. Як ETW пов'язує Sysmon (розділ 14.8) і EDR (розділ 14.10) технічно, і в чому полягає атака «ETW patching»?**

<details><summary>Відповідь</summary>

ETW — вбудований у ядро Windows механізм трасування подій; Sysmon сам технічно є одним із багатьох «споживачів» ETW-подій, так само як і EDR-платформи, що підписуються на десятки ETW-провайдерів для отримання телеметрії в реальному часі. «ETW patching» — техніка, за якою просунуті зловмисники частково відключають чи «патчать» функцію в userland-бібліотеці (`ntdll.dll`), відповідальну за генерацію ETW-подій, фактично осліплюючи і Sysmon, і EDR-агент, що покладаються виключно на цей канал — постійна гонка озброєнь між захисними механізмами й техніками ухилення.
</details>

**6. Чому «плоска» адміністративна модель робить NotPetya-подібний сценарій можливим, і як Tiered Administration Model це вирішує?**

<details><summary>Відповідь</summary>

У плоскій моделі один і той самий обліковий запис адміністратора використовується для входу на системи будь-якого рівня критичності — якщо облікові дані потрапляють у пам'ять звичайної, менш захищеної робочої станції (наприклад, через фішинг), зловмисник отримує прямий шлях до найкритичніших систем організації. Tiered Administration Model вирішує це незламним правилом: обліковий запис адміністратора певного рівня (Tier 0/1/2) ніколи не використовується для входу на машину нижчого рівня захисту — навіть повна компрометація Tier 2-робочої станції не дає доступу до облікових даних Tier 0.
</details>

**7. Чому Privileged Access Workstation (PAW) необхідна навіть за наявності Tiered Administration Model і Credential Guard?**

<details><summary>Відповідь</summary>

Tiered Administration Model визначає *правило* (які облікові дані де використовуються), а Credential Guard захищає *пам'ять* конкретної машини від дампінгу. Жоден із цих контролів не захищає від сценарію, коли той самий адміністратор, попри дотримання правил Tier, використовує ту саму фізичну робочу станцію для читання пошти й перегляду вебсторінок — один вдалий фішинговий лист компрометує машину, на якій згодом вводяться привілейовані облікові дані, ще до того, як Credential Guard встигне їх захистити (наприклад, через кейлогер чи інший механізм, що не залежить від дампінгу LSASS). PAW усуває саме цей ризик, фізично відокремлюючи адміністративну діяльність від повсякденних, ризикованих активностей.
</details>

## Підсумок модуля

Модуль 14 завершив спуск від управлінського рівня (Модуль 13) до найглибшого технічного шару — операційної системи хоста, на якій зрештою виконуються всі рішення попередніх модулів. CIS Benchmarks (14.2) дали методологічну основу; Group Policy та Linux-еквіваленти (14.3-14.6) — механізми масштабованого впровадження; патч-менеджмент і Configuration as Code (14.7) — підтримання цього стану в часі; журналювання й EDR (14.8-14.10) — видимість, коли профілактика все ж не спрацювала; Tiered Administration та PAW (14.11) — структурний захист найвищого рівня привілеїв від компрометації через найнижчий. Кожен із цих шарів окремо — недостатній; лише разом вони утворюють ту саму «остаточну лінію оборони», якою відкрився модуль.
