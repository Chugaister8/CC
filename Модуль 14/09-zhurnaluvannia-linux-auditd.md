# 14.9. Журналювання Linux: auditd та journald

## Дзеркальна пара для Linux-треку

Розділ 14.8 розглянув Windows-екосистему журналювання (Event Log + Sysmon). Linux має концептуально аналогічну, але технічно відмінну пару інструментів: **journald** для загального системного журналювання й **auditd** для деталізованого журналювання безпекових подій на рівні ядра.

## journald: сучасний централізований журнал

**journald**, частина systemd (стандартна система ініціалізації більшості сучасних дистрибутивів), збирає журнали від ядра, служб systemd і застосунків у єдиному бінарному, структурованому форматі (на відміну від класичних текстових файлів `/var/log/syslog`), доступному через `journalctl`:

```bash
# Перегляд журналів конкретної служби за останню годину
journalctl -u sshd --since "1 hour ago"

# Фільтрація за пріоритетом (лише помилки й критичні події)
journalctl -p err..alert

# Журнали конкретного користувача
journalctl _UID=1000
```

**Обмеження journald для безпекового аналізу:** подібно до стандартного Windows Event Log (розділ 14.8), journald фіксує загальні системні події й повідомлення служб, але не має вбудованого механізму для деталізованого, структурованого аудиту дій на рівні системних викликів (syscalls) — саме цю прогалину заповнює auditd, аналогічно ролі Sysmon у Windows-екосистемі.

## auditd: Linux Audit Framework

**auditd** — підсистема ядра Linux (Linux Audit Framework) для журналювання системних викликів, доступу до файлів та виконання команд на рівні, недосяжному стандартному журналюванню застосунків. Конфігурується через набір правил у `/etc/audit/rules.d/`:

```bash
# Приклади правил auditd (аналог Sysmon-конфігурації з розділу 14.8)

# Журналювання будь-яких змін файлу паролів
-w /etc/passwd -p wa -k identity_changes

# Журналювання будь-яких змін файлу sudoers
-w /etc/sudoers -p wa -k privilege_escalation_attempt

# Журналювання виконання команди sudo (доповнює власне логування sudo, розділ 14.5)
-a always,exit -F arch=b64 -S execve -F euid=0 -F auid!=0 -k root_command_execution

# Журналювання спроб завантаження модулів ядра (потенційний rootkit)
-w /sbin/insmod -p x -k kernel_module_loading
-w /sbin/modprobe -p x -k kernel_module_loading
```

Ключі `-k` (key) дозволяють групувати й швидко фільтрувати пов'язані події через `ausearch -k identity_changes`, аналогічно категоризації Sysmon-подій за Event ID у Windows.

> **Міні-вправа 14.9.1:** Правило auditd `-w /etc/passwd -p wa -k identity_changes` спрацьовує щоразу, коли будь-який процес намагається записати (`w`) чи змінити атрибути (`a`) файлу `/etc/passwd`. Адміністратор бачить у журналі спрацювання цього правила о 3-й ночі, ініційоване процесом, що не відповідає жодному запланованому завданню обслуговування. Яка послідовність подальших дій логічно випливає з практик, розглянутих у Модулі 12 (реагування на інциденти)?
>
> <details><summary>Відповідь</summary>
>
> Це класичний індикатор потенційної компрометації (IOC), що вимагає негайної тріаж-перевірки за IR Playbook-логікою з Модуля 07/12: перевірити, який саме процес і від імені якого користувача ініціював зміну (`ausearch -k identity_changes -ts today`), перевірити журнали автентифікації (`journald`/SSH-логи) на предмет підозрілого входу перед цим часом, перевірити фактичний вміст змін у `/etc/passwd` (чи додано новий, невідомий обліковий запис — техніка закріплення/persistence з Модуля 12, розділ 12.8), і за підтвердження компрометації — ескалувати за формальним процесом реагування на інциденти, а не просто відкоригувати файл і продовжити роботу без розслідування першопричини.
> </details>

## auditd проти journald: коли використовувати який

| | journald | auditd |
|---|---|---|
| Рівень деталізації | Повідомлення служб, загальні системні події | Системні виклики ядра, точні дії на рівні файлів/процесів |
| Продуктивність | Низький вплив | Вищий вплив при широких правилах (кожен syscall перевіряється проти набору правил) |
| Типове використання | Загальна діагностика, моніторинг доступності служб | Цілеспрямований аудит безпеки конкретних чутливих ресурсів |
| Аналогія з Windows-треком (14.8) | Windows Event Log (базовий рівень) | Sysmon (деталізована безпекова телеметрія) |

**Практична рекомендація:** широкі, невибіркові auditd-правила (наприклад, журналювання кожного системного виклику `open()` на всій файловій системі) створюють надлишковий обсяг даних і суттєве навантаження на продуктивність — так само, як неналаштований Sysmon генерує шум (розділ 14.8). Ефективна конфігурація цілеспрямовано журналює конкретні, безпеково значущі шляхи й дії (зміни `/etc/passwd`, `/etc/sudoers`, завантаження модулів ядра, виконання від імені root), а не намагається журналювати абсолютно все.

## Централізація: rsyslog та форвардинг у SIEM

Аналогічно Windows Event Forwarding (розділ 14.8), Linux-журнали (journald, auditd через `audisp` — audit dispatcher) типово пересилаються централізовано через `rsyslog` чи спеціалізовані агенти (Filebeat, Fluentd) на центральну SIEM-платформу. Без цієї централізації журнали, що залишаються лише локально на кожному сервері, втрачаються при компрометації самого сервера (зловмисник із достатніми привілеями може очистити локальні журнали, приховуючи сліди своєї активності) — централізоване, віддалене журналювання є прямим захистом проти цієї техніки anti-forensics.
