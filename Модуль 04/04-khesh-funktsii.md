# 4.4. Хеш-функції

Хеш-функція — це цифровий «відбиток пальця» для даних. Дайте їй будь-яку кількість даних — байт, гігабайт чи петабайт — і вона поверне фіксований рядок байт одного розміру. Змініть у документі одну літеру — і відбиток стане абсолютно іншим. Але з відбитка не можна відновити документ. Це принципово відрізняє хешування від шифрування: шифрування оборотне, хешування — ні. Саме ця «однобічність» робить хеш-функції незамінними для перевірки цілісності, зберігання паролів, цифрових підписів і блокчейну.

> 📖 Ключові терміни — у [глосарії модуля](00-glosariy.md).

## Властивості криптографічної хеш-функції

Не кожна хеш-функція є криптографічною. Криптографічна хеш-функція H повинна мати три ключових властивості:

### 1. Стійкість до прообразу (Preimage Resistance)

Маючи хеш h, неможливо знайти m, таке що H(m) = h.

«Однобічність» — основна властивість. Якщо зловмисник має хеш пароля, він не може обчислити сам пароль напряму.

### 2. Стійкість до другого прообразу (Second Preimage Resistance)

Маючи m₁, неможливо знайти m₂ ≠ m₁, таке що H(m₁) = H(m₂).

Зловмисник не може підмінити документ іншим з таким самим хешем.

### 3. Стійкість до колізій (Collision Resistance)

Неможливо знайти будь-яку пару m₁ ≠ m₂, таку що H(m₁) = H(m₂).

Ніхто не може знайти два різних документи з однаковим хешем — навіть якщо контролює обидва.

### Лавинний ефект (Avalanche Effect)

Зміна одного біта у вхідних даних має змінити приблизно половину бітів у хеші:

```python
import hashlib
msg1 = b"Кібербезпека"
msg2 = b"кібербезпека"  # лише регістр першої літери
print(hashlib.sha256(msg1).hexdigest())
print(hashlib.sha256(msg2).hexdigest())
# Виводи абсолютно різні — ніякої схожості
```

## Сімейства хеш-функцій: стан 2024 року

### MD5 — мертвий для безпекових застосувань

MD5 (1992) виробляє 128-бітний хеш. У 2004 році Ван Сяоюн продемонструвала практичні колізійні атаки. У 2008 році дослідники підробили SSL-сертифікат Certification Authority, використовуючи MD5-колізії. MD5 **не є криптографічно безпечним** — його не можна використовувати для перевірки цілісності, цифрових підписів або зберігання паролів.

**Де MD5 ще зустрічається (і чому це проблема):**
- Старі системи перевірки цілісності файлів.
- Застарілі бази паролів.
- Checksum в деяких протоколах (де безпека не потрібна, лише виявлення помилок — технічно допустимо, але краще замінити).

### SHA-1 — застарілий

SHA-1 (1995) виробляє 160-бітний хеш. Перша практична колізія продемонстрована Google у 2017 (SHAttered attack). Більшість браузерів і CA відмовились від SHA-1 у сертифікатах ще у 2017 році. **Не використовувати для нових систем.**

### SHA-2 — поточний стандарт

Сімейство SHA-2 (2001, NIST) є поточним стандартом для більшості застосувань:

| Алгоритм | Розмір хешу | Внутрішній стан | Застосування |
|---|---|---|---|
| SHA-224 | 224 біт | 256 біт | Рідко |
| **SHA-256** | 256 біт | 256 біт | TLS, підписи, Bitcoin PoW |
| SHA-384 | 384 біт | 512 біт | Вищий рівень безпеки |
| **SHA-512** | 512 біт | 512 біт | HMAC, підписи, сховища паролів |
| SHA-512/256 | 256 біт | 512 біт | Швидший за SHA-256 на 64-бітних |

SHA-256 — найпоширеніший вибір; SHA-384/512 — для вищого рівня безпеки або де потрібна більша колізійна стійкість.

### SHA-3 — резервний стандарт

SHA-3 (2015) — результат конкурсу NIST (2007–2012). Переміг Keccak — принципово інша архітектура (sponge construction), на відміну від SHA-2 (Merkle-Damgård). SHA-3 є страховкою: якщо буде знайдена фундаментальна слабкість у Merkle-Damgård (що вразить SHA-2), SHA-3 залишиться безпечним.

**SHAKE128/SHAKE256** — «розтяжні» варіанти SHA-3 (XOF — extendable output function), що дозволяють задати довільний розмір виходу.

### BLAKE2 і BLAKE3 — швидкі альтернативи

**BLAKE2** (2012) — швидша за SHA-256 і SHA-3 при збереженні безпеки; використовується в Wireguard, libsodium, деяких Argon2-реалізаціях.

**BLAKE3** (2020) — ще швидша, паралельна архітектура; молодша і менш перевірена часом, але вже використовується в ряді систем.

## HMAC: автентифікація з хешем

**HMAC (Hash-based Message Authentication Code)** — використовує хеш-функцію разом з ключем для автентифікації повідомлення:

```
HMAC(K, m) = H((K ⊕ opad) || H((K ⊕ ipad) || m))
```

- Надає **автентифікацію** (тільки той, хто має ключ K, може обчислити вірний HMAC) і **цілісність** (зміна m дає інший HMAC).
- Захищений від length extension attacks — проблеми наївного `H(K || m)`.

**HMAC-SHA256 — стандартний вибір** для автентифікації API-запитів, JWT-токенів, перевірки webhook.

```python
import hmac, hashlib

key = b"secret_api_key"
message = b"order_id=12345&amount=100&currency=UAH"

mac = hmac.new(key, message, hashlib.sha256).hexdigest()
print(f"HMAC-SHA256: {mac}")

# Верифікація (constant-time порівняння — важливо!)
def verify_hmac(key, message, received_mac):
    expected = hmac.new(key, message, hashlib.sha256).digest()
    received = bytes.fromhex(received_mac)
    return hmac.compare_digest(expected, received)  # НЕ == , бо timing attack
```

> **Чому `hmac.compare_digest`, а не `==`?** Звичайне порівняння рядків зупиняється на першому відмінному байті — це timing attack (розділ 4.8): зловмисник може виміряти час порівняння і по одному байту відгадати вірний MAC. `compare_digest` завжди порівнює всі байти за однаковий час.

## Хешування паролів: окремий клас алгоритмів

Це найважливіший практичний підрозділ розділу для більшості розробників.

**Чому SHA-256 не підходить для паролів?** SHA-256 розроблений, щоб бути **швидким**: сучасна GPU обчислює мільярди SHA-256 за секунду. Це ідеально для підпису файлів, але катастрофічно для паролів: зловмисник, що отримав базу хешів SHA-256, може перебирати словники і мільярди комбінацій із величезною швидкістю.

**Правильні алгоритми хешування паролів** навмисно **повільні** і використовують велику кількість пам'яті:

| Алгоритм | Рік | Пам'ять | Паралельність | Рекомендація |
|---|---|---|---|---|
| **bcrypt** | 1999 | Низька (~4 КБ) | Ні | ✅ Надійний, перевірений |
| **scrypt** | 2009 | Висока (налаштовна) | Ні | ✅ Стійкий до GPU/ASIC |
| **Argon2id** | 2015 | Висока (налаштовна) | Так | ✅ Переможець PHC, рекомендований NIST |
| PBKDF2 | 2000 | Низька | Так | ⚠️ Слабший за нові, але прийнятний |
| MD5/SHA-1/SHA-256 | — | — | — | ❌ Ніколи для паролів |

### Argon2id: сучасний стандарт

Argon2 (2015) виграв Password Hashing Competition і рекомендований NIST SP 800-63B. Три варіанти:
- **Argon2d** — стійкий до GPU (але вразливий до side-channel).
- **Argon2i** — стійкий до side-channel (але менш до GPU).
- **Argon2id** — гібрид; рекомендований для більшості застосувань.

```python
# pip install argon2-cffi
from argon2 import PasswordHasher

ph = PasswordHasher(
    time_cost=3,        # кількість ітерацій
    memory_cost=65536,  # 64 МБ пам'яті
    parallelism=4,      # кількість паралельних потоків
    hash_len=32,
    salt_len=16
)

# Хешування (salt генерується автоматично)
hash_val = ph.hash("MyP@ssw0rd123!")
print(hash_val)  # $argon2id$v=19$m=65536,t=3,p=4$...

# Верифікація
try:
    ph.verify(hash_val, "MyP@ssw0rd123!")
    print("Пароль вірний")
except Exception:
    print("Невірний пароль")
```

### Важливість salt при хешуванні паролів

**Salt** — унікальний випадковий рядок, що додається до пароля перед хешуванням і зберігається разом з хешем. Це гарантує, що два однакових паролі мають різні хеші:

```
hash("password123" + "xK9mL2") = A1B2C3...  ← для User1
hash("password123" + "pQ7nR5") = D4E5F6...  ← для User2
```

Без salt: зловмисник обчислює хеш `password123` один раз і знаходить всіх користувачів з цим паролем одночасно. З salt: для кожного запису потрібен окремий підбір.

Сучасні алгоритми (Argon2, bcrypt, scrypt) генерують і зберігають salt автоматично — ви не мусите керувати ним вручну.

## Практичні застосування хеш-функцій

- **Перевірка цілісності файлів** — SHA-256 суми при завантаженні (перевірте! це захищає від підміни файлів).
- **Цифрові підписи** — підписується хеш документа, а не сам документ (швидше і розмір підпису фіксований).
- **Merkle Tree** — блокчейн, git — структура, де хеш батьківського вузла залежить від хешів дочірніх; зміна будь-якого листка змінює весь корінь.
- **Fingerprint сертифікатів** — SHA-256 відбиток сертифіката для швидкої ідентифікації.
- **Deduplicate storage** — однакові файли мають однаковий хеш → зберігається один примірник.
- **Password storage** — Argon2id/bcrypt/scrypt (але не SHA-256/MD5).

## Міні-вправа

```python
import hashlib

# Продемонструвати лавинний ефект
msgs = [b"hello", b"hello!", b"Hello", b"hello world"]
for m in msgs:
    h = hashlib.sha256(m).hexdigest()
    print(f"{str(m):<15} → {h[:32]}...")
```
