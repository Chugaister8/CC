# 4.9. Українські криптографічні стандарти

Більшість посібників з криптографії зосереджені на NIST-стандартах і не згадують, що численні країни мають власні криптографічні стандарти. Але якщо ви розробляєте систему для українського держоргану, подаєте документи через «Дію» або інтегруєтесь з ДПС — ви вже взаємодієте з криптографічними алгоритмами, що тут описані, навіть не знаючи про це. КЕП, яким підписаний кожен документ, що ви надсилаєте до держреєстрів, — це ДСТУ 4145. Хеш у структурі цього підпису — це Купина. Знати ці стандарти — не академічна деталь, а практична необхідність для українського IT-фахівця. Та й з чисто криптографічної точки зору: «Купина» і «Калина» — сучасні, добре спроектовані алгоритми, розроблені командою ХНУРЕ і КПІ та визнані академічною спільнотою.

> 📖 Ключові терміни — у [глосарії модуля](00-glosariy.md).

## Контекст: навіщо власні стандарти

**Суверенний контроль:** держава, що покладається виключно на іноземні стандарти, теоретично залежна від можливих «backdoor» або слабкостей, відомих лише розробнику стандарту. Навіть якщо AES не має прихованих вразливостей (і немає підстав вважати, що є), ця залежність є стратегічним ризиком.

**Правова вимога:** ЗУ «Про захист інформації в інформаційно-комунікаційних системах» і підзаконні акти встановлюють вимоги щодо криптографічного захисту державної інформації з обмеженим доступом.

**Технічна незалежність у воєнний час:** після 2022 року значення власної криптографічної бази різко зросло. Навіть для несекретних систем критичної інфраструктури перехід на вітчизняні стандарти є рекомендованим напрямком розвитку.

## ДСТУ 7564:2014 — Хеш-функція «Купина»

**«Купина» (Kupyna)** — українська стандартна хеш-функція, розроблена командою фахівців Харківського національного університету радіоелектроніки (НТУ «ХПІ», ХНУРЕ) і затверджена як ДСТУ 7564:2014.

### Технічні характеристики

**Архітектура:** sponge construction (як SHA-3/Keccak), але з власним дизайном — два варіанти внутрішнього блоку:

| Варіант | Розмір виходу | Розмір блоку | Аналог NIST |
|---|---|---|---|
| **Kupyna-256** | 256 біт | 512 біт | SHA-256 |
| **Kupyna-384** | 384 біт | 1024 біт | SHA-384 |
| **Kupyna-512** | 512 біт | 1024 біт | SHA-512 |

**Раунди:** 10 раундів для 512-бітного блоку, 14 раундів для 1024-бітного.

**Внутрішні операції кожного раунду:**
- **AddRoundConstant** — XOR з константою раунду (захист від слабких ключів).
- **SubBytes** — нелінійна підстановка через S-box (аналог AES SubBytes).
- **ShiftBytes** — зсув рядків стану (аналог AES ShiftRows).
- **MixColumns** — лінійне перемішування стовпців (аналог AES MixColumns у GF(2⁸)).

Архітектура свідомо схожа на AES — це дозволяє ефективне апаратне прискорення на AES-NI інструкціях.

### Криптографічний аналіз

Купина пройшла через відкритий криптоаналіз міжнародною спільнотою. На дату публікації посібника не виявлено жодних практичних атак на повну версію. Алгоритм прийнятий як стандарт низкою міжнародних органів стандартизації та рекомендований ENISA для використання в держсекторі.

### Практичне використання

```python
# Kupyna не входить у стандартну бібліотеку Python
# Варіанти: pyKupyna або реалізації на основі C-бібліотек

# Приклад через ctypes (якщо є C-реалізація libkupyna)
# або через офіційні бібліотеки АЦСК

# Також: OpenSSL з патчами від ДСССЗІ підтримує Купину
# openssl dgst -kupyna256 file.txt

# Довідковий Python (для демонстрації, не production):
# pip install kupyna (сторонній пакет, перевіряйте довіру)
```

Купина **обов'язкова** для:
- Кваліфікованого електронного підпису (КЕП) на основі ДСТУ 4145.
- Захисту державної таємниці і конфіденційної інформації держорганів.
- Систем документообігу, що підключаються до держреєстрів.

## ДСТУ 7624:2014 — Блоковий шифр «Калина»

**«Калина» (Kalyna)** — українська стандартна блокова функція шифрування, затверджена як ДСТУ 7624:2014. Розроблена тими самими фахівцями ХНУРЕ/КПІ, що і Купина.

### Технічні характеристики

| Варіант | Розмір ключа | Розмір блоку | Кількість раундів |
|---|---|---|---|
| **Kalyna-128/128** | 128 біт | 128 біт | 10 |
| **Kalyna-256/128** | 256 біт | 128 біт | 14 |
| **Kalyna-256/256** | 256 біт | 256 біт | 14 |
| **Kalyna-512/256** | 512 біт | 256 біт | 18 |
| **Kalyna-512/512** | 512 біт | 512 біт | 18 |

**Структура:** SPN (Substitution-Permutation Network) — аналогічна AES, але з розширеними блоками і більшою кількістю раундів.

**Ключові операції:**
- SubBytes через GF(2⁸) S-box.
- ShiftRows (налаштований під більші розміри блоку).
- MixColumns через MDS-матрицю.
- XOR з ключами раундів.

**Порівняння з AES:**

| Критерій | AES-256 | Kalyna-512/512 |
|---|---|---|
| Розмір ключа | 256 біт | 512 біт |
| Розмір блоку | 128 біт | 512 біт |
| Кількість раундів | 14 | 18 |
| Швидкість (software) | Висока | Дещо нижча |
| Апаратне прискорення | AES-NI (широко) | Менш поширено |

### Режими роботи для Калини

ДСТУ 7624 визначає кілька режимів роботи (аналоги CBC, CTR, GCM):
- **ECB** — для окремих блоків (не для потоків даних).
- **CBC, CFB, OFB** — режими роботи з зворотним зв'язком.
- **CTR** — потоковий режим.
- **GCM, CCM** — аутентифіковане шифрування (AEAD).

## ДСТУ 4145-2002 — Цифровий підпис на еліптичних кривих

**ДСТУ 4145** (редакція 2002, оновлення 2010) — стандарт цифрового підпису на основі еліптичних кривих (аналог ECDSA).

**Особливості:**
- Визначає власні параметри еліптичних кривих для різних рівнів безпеки (163, 167, 173, 179, 191, 210 біт).
- Основний механізм для **КЕП (Кваліфікованого Електронного Підпису)** в Україні.
- Підтримується всіма АЦСК і системами документообігу з держорганами.

**Правова основа:** ЗУ «Про електронні довірчі послуги» (2017) визначає КЕП на основі ДСТУ 4145 і Купини як юридично рівнозначний власноручному підпису.

## Нормативна база

| Нормативний акт | Що регулює |
|---|---|
| ДСТУ 7564:2014 | Хеш-функція «Купина» |
| ДСТУ 7624:2014 | Блоковий шифр «Калина» |
| ДСТУ 4145-2002 | Цифровий підпис на ЕК |
| ЗУ «Про електронні довірчі послуги» | КЕП, АЦСК, правова сила |
| ЗУ «Про захист інформації в ІКС» | Обов'язковий захист держінформації |
| Постанова КМУ № 693/2013 | Вимоги до КСЗІ |

**ДСССЗІ (Державна служба спеціального зв'язку та захисту інформації)** — регулятор у сфері технічного захисту інформації і криптографії в Україні. Видає ліцензії, сертифікує засоби криптографічного захисту (ЗКЗІ).

## Де і як використовуються українські стандарти

**Обов'язково:**
- Криптографічний захист державної таємниці.
- КЕП для взаємодії з держорганами (ДПС, ПФУ, Мін'юст, Пенсійний фонд).
- Інформаційні системи з обмеженим доступом у держорганах.
- Системи захисту критичної інфраструктури.

**Що це означає для розробника:** якщо ваш продукт інтегрується з API «Дії», системою електронних торгів (ProZorro), Єдиним державним реєстром або будь-яким держреєстром — ваш код вже зараз перевіряє або генерує підписи за ДСТУ 4145 з хешуванням Купиною. Переконайтесь, що SDK або бібліотека, яку ви використовуєте, дійсно реалізує ДСТУ, а не лише «схожий» ECDSA.

**Рекомендовано (не обов'язково для комерційних систем):**
- Корпоративні системи, що обмінюються даними з держорганами.
- Фінансові установи, що мають ліцензії від НБУ.

**Не потрібно (для більшості комерційних веброзробок):**
- Звичайні вебсайти, мобільні застосунки без держінтеграцій.
- Системи, що не зберігають держдані або персональні дані у великому масштабі.

## Практична реалізація

```python
# Для взаємодії з КЕП в Python
# Офіційні АЦСК надають SDK або приклади на Python/Java/.NET

# Перевірка КЕП через API IIT (iit.com.ua) або CЗО Sign Widget

# Генерація тестового КЕП для розробки:
# Дія (diia.gov.ua) → Документи → Цифровий підпис
# Або через Приватбанк: Приват24 → Послуги → ЕЦП

# Для тестової верифікації підпису:
# Мінцифри надає публічний сервіс верифікації: czo.gov.ua
```

## Міні-вправа

1. Зайдіть на `czo.gov.ua` (Центр Забезпечення Операцій) і завантажте будь-який публічний документ з КЕП (наприклад, офіційний акт держоргану). Перевірте підпис через сервіс верифікації. Які алгоритми використані?
2. Знайдіть у своєму телефоні або email один з КЕП-підписаних документів (квитанція від ДПС, витяг з реєстру). Відкрийте файл `.p7s` або `.asic` у text-редакторі і спробуйте знайти OID алгоритму підпису. ДСТУ 4145 має власний OID: `1.2.804.2.1.1.1.1.3`.
