# 9.11. Чек-лист і самоперевірка

## Підсумок модуля: ключові ідеї

1. **Shared Responsibility Model — фундамент хмарної безпеки.** Провайдер захищає «хмару», клієнт захищає «у хмарі» — конкретний розподіл залежить від IaaS/PaaS/SaaS.
2. **Мережева ізоляція у хмарі гнучкіша, але вимагає усвідомленого дизайну.** VPC, Security Groups, Private Subnets — інструменти не захищають за замовчуванням, потребують правильної конфігурації.
3. **IAM — найважливіший контроль хмарної безпеки.** Більшість хмарних інцидентів пов'язані з надмірними правами або вкраденими credentials, а не з вразливостями інфраструктури.
4. **Шифрування і управління секретами — необхідність, не опція.** KMS, Secrets Manager, правильна конфігурація S3 — базовий захист даних.
5. **Контейнери не ізольовані так, як VM.** Спільне ядро ОС, потреба в Pod Security Standards, Network Policies, image scanning.
6. **DevSecOps інтегрує безпеку в CI/CD, а не додає її після.** Shift Left економічно ефективніший за виявлення проблем у продакшені.
7. **CSPM і CWPP вирішують різні задачі.** CSPM перевіряє конфігурацію; CWPP захищає workload під час виконання.
8. **Сертифікація провайдера ≠ ваша відповідність.** SOC 2/ISO 27001 хмарного провайдера покриває нижні рівні SRM; ваша частина відповідальності залишається вашою.
9. **SSRF → Instance Metadata — підручниковий хмарний вектор атаки.** Capital One (2019) демонструє, як вебвразливість конвертується у повну компрометацію хмарного акаунту через IMDS.

## Орієнтовні відповіді

<details>
<summary>Розгорнути відповіді</summary>

1. IaaS: клієнт патчує ОС самостійно (повний контроль і повна відповідальність). PaaS: провайдер патчує ОС і runtime, клієнт лише код. SaaS: провайдер відповідає за все нижче застосунку, включаючи ОС.

2. Microsoft надав widely-scoped SAS token замість URL-specific, що дозволило доступ до 38 ТБ внутрішніх даних. Технічно Microsoft (провайдер) забезпечив інструмент (SAS tokens) правильно, але команда (клієнт цього ж інструменту всередині Microsoft) неправильно його сконфігурувала — показує, що навіть провайдер сам є "клієнтом" своїх інструментів і підпадає під ту саму логіку SRM.

3. Configuration Drift: початкова конфігурація безпечна, але з часом змінюється через ручні правки, hotfix під тиском дедлайнів, тимчасові правила, що стають постійними. Хронічна проблема тому, що хмарна інфраструктура змінюється швидше за традиційну, а контроль за змінами часто послаблений заради швидкості DevOps.

4. Security Groups: stateful — якщо дозволено вхідне з'єднання, відповідь автоматично дозволена без окремого правила. NACLs: stateless — потрібні явні правила для обох напрямків (вхідний і вихідний) окремо.

5. NAT Gateway дозволяє ресурсам у Private Subnet (без публічної IP) ініціювати вихідні з'єднання (завантажити оновлення, викликати зовнішній API), але блокує будь-які вхідні з'єднання ззовні. Це дає доступ в інтернет без прямої публічної експозиції ресурсу.

6. Rate Limiting обмежує кількість запитів з однієї IP за період часу. Проти DDoS: зупиняє volumetric flood з окремих джерел (хоча не захищає від розподілених тисяч IP). Проти brute force: обмежує кількість спроб логіну з однієї IP, роблячи перебір паролів неефективним.

7. VPC Flow Logs записують метадані кожного IP-пакету (джерело, призначення, порт, протокол, ACCEPT/REJECT). Аналіз REJECT-записів: послідовні спроби підключення до різних портів з одного джерела = ознака port scanning; типовий сигнал розвідки перед атакою.

8. IAM Roles надають тимчасові credentials, що автоматично ротуються і ніколи не з'являються у статичному вигляді в коді чи конфіг-файлах. IAM Users з access keys — статичні credentials, що можуть витекти через git, логи, конфіг-файли і залишатись дійсними роками без ротації.

9. Permission Boundary встановлює максимальну стелю дозволів незалежно від того, що дозволяє звичайна Policy. Якщо Policy дає AdministratorAccess, але Boundary обмежує до S3ReadOnly — ефективний результат буде S3ReadOnly. Це другий незалежний шар обмеження, корисний для делегування створення IAM ресурсів без ризику ескалації.

10. Workload Identity Federation дозволяє зовнішнім системам (GitHub Actions, інший хмарний провайдер) отримувати тимчасові хмарні credentials через federation замість статичних service account keys. Вирішує проблему: статичні ключі в CI/CD конфігурації — типове джерело витоку; federation усуває потребу зберігати їх взагалі.

11. Cross-Account Role дозволяє ресурсам одного AWS акаунту отримати тимчасовий доступ до ресурсів іншого через AssumeRole. ExternalId у Trust Policy захищає від Confused Deputy Problem: третя сторона (наприклад, SaaS-провайдер), що має доступ до вашого акаунту через роль, не може бути обманута іншим клієнтом цього ж провайдера видати себе за вас, бо ExternalId унікальний для кожного клієнта.

12. RDS encryption застосовується на рівні фізичного зберігання при створенні екземпляра. Зміна шифрування для існуючих даних потребує повного re-encryption всіх блоків — AWS не підтримує "on-the-fly" зміну для існуючого instance; єдиний шлях: snapshot → новий зашифрований instance з відновленого snapshot → перенесення трафіку.

13. Envelope Encryption: KMS генерує Data Encryption Key (DEK), DEK шифрує великі обсяги даних локально (швидко), а сам DEK зберігається у зашифрованому вигляді (encrypted DEK) поруч з даними. Замість прямого виклику KMS для кожного байту даних (повільно, дорого) — KMS викликається лише раз для шифрування/розшифрування невеликого DEK.

14. Block Public Access (account+bucket level), Bucket Policy з deny не-HTTPS, Versioning увімкнено (захист від ransomware), MFA Delete для критичних бакетів, Object Lock (WORM) для compliance-вимог.

15. AWS Macie — ML-сервіс класифікації чутливих даних у S3. Виявляє: номери кредитних карток (PCI), SSN (для США), паспортні дані, медичні ідентифікатори, AWS Access Keys у файлах, email-адреси, телефони, IP-адреси.

16. Контейнери ділять одне ядро ОС хоста (через Namespaces і Cgroups для ізоляції), на відміну від VM, де гіпервізор забезпечує повну апаратну ізоляцію кожної гостьової ОС. Вразливість ядра Linux (наприклад, Dirty COW) може дозволити container escape — вихід за межі контейнера на хост-рівень, що неможливо для коректно ізольованих VM.

17. Pod Security Standards: Privileged (без обмежень, для системних компонентів), Baseline (запобігає відомим ескалаціям привілеїв), Restricted (найсуворіший, поточна найкраща практика — заборона privileged, hostPath, root за замовчуванням).

18. Kubernetes Secrets за замовчуванням зберігаються у etcd лише у base64-encoded форматі — не зашифровані. Base64 легко декодується (`base64 -d`), це не криптографічний захист, лише кодування. Потребує додаткового encryption at rest для etcd або зовнішнього secrets manager (External Secrets Operator) для реального захисту.

19. Default Deny Network Policy блокує весь трафік до/від Pod'ів, доки не визначені явні allow-правила. Рекомендована практика тому, що за замовчуванням Kubernetes дозволяє весь трафік між усіма Pod'ами в кластері — у багатоорендних або складних кластерах це означає, що компрометація одного Pod дає мережевий доступ до всіх інших.

20. Shift Left: вартість виправлення дефекту зростає експоненційно з кожним етапом життєвого циклу розробки (IDE → Code Review → CI → Staging → Production), за оцінками IBM різниця може сягати 100x між IDE і Production. Виявлення проблем якомога раніше — найдешевший спосіб їх усунути.

21. SAST аналізує вихідний код вашого застосунку на патерни вразливостей (SQL injection, hardcoded secrets, insecure crypto). SCA аналізує сторонні залежності (бібліотеки, пакети) на відомі CVE у конкретних версіях. Різні об'єкти аналізу: ваш код vs чужий код, який ви використовуєте.

22. SLSA (Supply chain Levels for Software Artifacts): SLSA 1 — задокументований build process; SLSA 2 — версіонований source control + build service; SLSA 3 — hardened build platform з non-falsifiable provenance; SLSA 4 — two-party review, hermetic, reproducible builds. Вищий рівень = вища довіра до того, що артефакт не скомпрометований у процесі збірки.

23. Pre-commit hook зупиняє секрет ДО потрапляння в git history взагалі. CI-перевірка виявляє секрет вже ПІСЛЯ коміту і push — секрет вже в історії git (навіть якщо видалений у наступному коміті, він залишається доступним через `git log`). Pre-commit запобігає проблемі, CI лише її виявляє постфактум.

24. CSPM відповідає на питання "Чи правильно НАЛАШТОВАНО хмарне середовище?" — перевіряє конфігурацію статично. CWPP відповідає на питання "Що відбувається ВСЕРЕДИНІ workload ЗАРАЗ?" — моніторить поведінку в реальному часі під час виконання.

25. CIEM (Cloud Infrastructure Entitlement Management) спеціалізується на аналізі надмірних IAM-прав і entitlements — виявляє права, які надані, але фактично не використовуються (Least Privilege Recommendations), і потенційні шляхи ескалації привілеїв через комбінації дозволів.

26. Приклад CRITICAL знахідки: S3 bucket публічно доступний для запису; Security Group дозволяє 0.0.0.0/0 на порт 22 (SSH); IAM user з програмним доступом і AdministratorAccess; root account без MFA.

27. Type I оцінює дизайн контролів на конкретну дату (момент часу) — простіше і швидше отримати. Type II оцінює ефективність контролів протягом періоду (зазвичай 6-12 місяців) — потребує реального функціонування контролів протягом тривалого часу і значно складніший та цінніший для отримання (підтверджує, що контролі дійсно працюють, а не лише задокументовані).

28. Schrems II поставив під юридичний сумнів передачу персональних даних ЄС-громадян у США через хмарні сервіси (через відсутність "адекватного" захисту від доступу спецслужб США). У відповідь AWS, Azure, GCP запровадили EU-специфічні регіони та "Sovereign Cloud" рішення, що тримають дані фізично і юридично в межах ЄС.

29. DPA — Data Processing Agreement: юридична угода між Data Controller (ваша компанія) і Data Processor (хмарний провайдер), що визначає умови обробки персональних даних відповідно до GDPR Art. 28. Потрібен завжди, коли хмарний сервіс обробляє персональні дані ЄС-громадян від імені вашої компанії.

30. Well-Architected Review — структурована оцінка архітектури проти найкращих практик. Шість стовпів: Operational Excellence, Security, Reliability, Performance Efficiency, Cost Optimization, Sustainability. Security Pillar фокусується на: управлінні credentials, виявленні і реагуванні на інциденти, захисті даних, захисті мережі і compute.

31. Capital One: 1) Вебзастосунок Capital One мав SSRF-вразливість у WAF-конфігурації (можливість передати довільний URL). 2) Зловмисник надіслав запит до внутрішнього сервера з URL `http://169.254.169.254/...`. 3) Сервер (виконуючи запит від імені застосунку) звернувся до Instance Metadata Service. 4) IMDS повернув тимчасові IAM credentials ролі EC2-інстанції. 5) Зловмисник використав ці credentials для доступу до S3 і викрав 100 млн записів клієнтів.

32. IMDSv2 вимагає спочатку отримати session token через PUT-запит з спеціальним заголовком, перед тим як можна виконати GET-запит до metadata endpoints. Проста SSRF-вразливість зазвичай дозволяє лише підставити URL у GET-запит (як у Capital One сценарії) — вона технічно не може виконати PUT-запит з потрібними заголовками, тому атака через просту SSRF на IMDSv2 не спрацьовує.

33. IAM Privilege Escalation Chain: послідовність здавалося б обмежених дозволів, що в комбінації дозволяють отримати ширші права. Приклад: `iam:CreatePolicyVersion` (модифікувати існуючу policy) + `iam:PassRole` (передати роль сервісу) + здатність запустити Lambda/EC2 → можна модифікувати policy, прикріплену до ролі з ширшими правами, потім запустити сервіс з цією роллю.

34. Хмарні credentials дають доступ до compute resources (EC2, GPU instances), що коштують гроші власнику акаунту незалежно від того, чи зловмисник краде дані. Запуск crypto-майнінгу — найшвидший спосіб монетизувати вкрадені credentials без потреби продавати викрадені дані на чорному ринку: рахунок генерується миттєво і безпосередньо, поки власник акаунту не помітить.

35. Базовий CIS AWS аудит перевіряє: Root MFA, невикористовувані credentials (>90 днів), надмірно широкі IAM policies (Action:* Resource:*), відкриті Security Groups на чутливі порти (22, 3389), S3 Public Access Block конфігурацію, S3 default encryption, CloudTrail multi-region покриття і log file validation.

36. S3 bucket enumeration можливий тому, що S3 bucket назви глобально унікальні і доступні через прогнозований URL-патерн (`bucketname.s3.amazonaws.com`). Якщо Block Public Access не налаштовано, прямий HTTP GET повертає 200 OK з лістингом вмісту без жодної автентифікації. Небезпечно тому, що зловмисники використовують саме такі автоматизовані скрипти для масового сканування типових назв бакетів (company-backup, company-data тощо) у пошуку випадково відкритих сховищ.

37. "Червоні прапори" приховування слідів у CloudTrail: `StopLogging` (зупинка журналювання), `DeleteTrail` (видалення CloudTrail), `DeleteFlowLogs` (видалення VPC Flow Logs), особливо якщо ці події відбуваються відразу перед або одночасно з підозрілою активністю (створення нових users, прикріплення адмін-policy).

38. Microsoft AI SAS token incident (2023) — 38 ТБ внутрішніх даних через широкий SAS-токен; Capital One breach (2019) — 100 млн записів через SSRF→IMDS; Industroyer/Industroyer2 (хоча технічно OT, не cloud) і атаки на українські хмарні Microsoft 365/Google Workspace акаунти через фішинг (з модуля 05/07).

39. "Ми в AWS, отже ми SOC 2 compliant" хибне, бо SOC 2 сертифікація AWS покриває лише нижні рівні Shared Responsibility Model (фізична безпека, мережева інфраструктура, гіпервізор). Ваша частина відповідальності — конфігурація IAM, шифрування ваших даних, контроль доступу до ваших застосунків — це повністю ваша відповідальність і вимагає окремого аудиту вашої частини системи, незалежно від сертифікації провайдера.

40. Три практики, застосовні універсально: 1) MFA для всіх привілейованих/адміністративних акаунтів; 2) Централізоване логування всіх адміністративних дій (CloudTrail/Activity Log/Cloud Audit Logs) з multi-region/multi-resource покриттям; 3) Принцип найменших привілеїв для IAM/RBAC з регулярним аудитом невикористовуваних прав.

</details>
