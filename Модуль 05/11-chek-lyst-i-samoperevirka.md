# 5.11. Чек-лист і самоперевірка

## Підсумок модуля: ключові ідеї

1. **Ідентифікація, автентифікація і авторизація — три різних операції.** Плутанина між ними призводить до конкретних вразливостей: IDOR, JWT без перевірки підпису, авторизація без автентифікації.

2. **Пароль — найслабший фактор.** Credential stuffing, password spraying, phishing — все це обходить навіть «складний» пароль без MFA. MFA обов'язкова; FIDO2/Passkeys — єдиний метод, захищений від фішингу.

3. **Парольний менеджер — не опція, а необхідність.** Унікальний пароль для кожного сервісу + перевірка на HIBP — єдина реалістична відповідь на credential stuffing.

4. **SSO, SAML і OAuth вирішують одну задачу — федерацію** — але по-різному і з різними ризиками. SAML — enterprise корпоративний, OIDC — сучасний consumer, OAuth — делегована авторизація (не автентифікація!).

5. **RBAC — найпоширеніша і одна з найбільш «зламаних» моделей.** Role explosion, creeping privileges, відсутність SoD — типові проблеми. ABAC гнучкіший, але складніший.

6. **Привілейовані акаунти потребують окремої стратегії.** LAPS для локальних адмінів, gMSA для сервісних акаунтів, JIT замість standing privileges, session recording.

7. **Zero Trust — не продукт, а архітектурна парадигма.** Безперервна верифікація, мікросегментація, ризик-адаптивний контроль, ZTNA замість VPN.

8. **Kerberoasting, Golden Ticket, Pass-the-Hash** — не теоретичні атаки. Credential Guard, AES замість RC4, gMSA, LAPS, моніторинг Event ID — конкретні контрзаходи.

## Практичний чек-лист

### Для особистого використання

- [ ] Парольний менеджер встановлено і використовується для всіх акаунтів.
- [ ] Кожен важливий акаунт має унікальний пароль (перевірено менеджером).
- [ ] MFA увімкнена для email, банківських акаунтів, корпоративних систем.
- [ ] Метод MFA: TOTP або FIDO2 (не лише SMS).
- [ ] Основний email перевірено на HIBP.
- [ ] Переглянуті авторизовані OAuth-застосунки (Google, Microsoft).
- [ ] Пасткі/відновлювальні коди MFA збережені в безпечному місці.

### Для організації

| Захід | Пріоритет | Перевірка |
|---|---|---|
| MFA для всіх users (не лише адмінів) | Must | Conditional Access / IdP policy |
| FIDO2 або Passkey для адміністраторів | Must | Hardware key inventory |
| LAPS встановлено | Must | Get-LapsADPassword |
| Сервісні акаунти переведені на gMSA | Must | Get-ADServiceAccount |
| JIT-доступ для привілейованих операцій | Must | PIM / PAM рішення |
| Legacy Auth (Basic Auth) вимкнено | Must | Azure AD Sign-in logs |
| SoD для критичних бізнес-процесів | Must | IAM Review |
| Аудит прав кожні 90 днів | Must | Access Review |
| Credential Guard увімкнено | Should | Get-CimInstance DeviceGuard |
| SCIM для деprovisioning | Should | Перевірка акаунтів звільнених |
| Session recording для PAM | Should | PAM рішення |
| UEBA/Anomaly detection | Should | SIEM |
| Zero Trust Network Access | Should | ZTNA рішення |
| Реакційний план на компрометацію облікових даних | Must | IRP документ |

## Орієнтовні відповіді

<details>
<summary>Розгорнути відповіді</summary>

1. Вхід у банк: Ідентифікація — введення номера картки/логіну (заявка «я — клієнт 12345»). Автентифікація — введення пароля/PIN (доведення заявки). Авторизація — система перевіряє: чи може клієнт 12345 переказати суму X з рахунку Y (перевірка прав на дію).
2. Creeping privileges: користувач переходить між відділами, отримуючи нові права, але старі не відкликаються. Через роки він має права всіх колишніх посад — значно ширші, ніж потрібні зараз. Небезпечно: надмірний доступ, що може бути використаний або скомпрометований.
3. Leaver — найкритичніший: якщо акаунт звільненого залишається активним — колишній співробітник або зловмисник з його обліковими даними зберігає доступ до корпоративних ресурсів. Автоматизований deprovisioning через SCIM і чіткий процес — критично.
4. IdP (Identity Provider) — зберігає і верифікує ідентичності (Azure AD, Google Workspace). SP/RP (Service Provider / Relying Party) — надає сервіс і довіряє IdP для автентифікації (Salesforce, GitHub). SP не зберігає паролі користувачів.
5. Знання (пароль, PIN); Володіння (TOTP-застосунок, YubiKey, смарт-карта); Ознака (відбиток, обличчя, голос); Місце (IP-адреса, мережа); Поведінка (динаміка набору, хода).
6. Пароль і PIN — обидва є чимось, що «знаєш». MFA потребує двох факторів **різних категорій**. Знання + знання = посилена однофакторна автентифікація, але не MFA.
7. FIDO2: підпис прив'язаний до RPID (домену сайту). На фішинговому `evil-example.com` аутентифікатор підписує `evil-example.com`, а не `example.com` → сервер відхиляє. TOTP: 6-значний код не прив'язаний до домену → зловмисник перехоплює код у реальному часі і використовує його на справжньому сайті протягом 30 секунд.
8. MFA Fatigue: зловмисник надсилає нескінченні push-запити, поки жертва випадково натискає «Так». Захист: Number Matching (ввести число з push в застосунок — не просто натиснути) або FIDO2 (не використовує push взагалі).
9. Passkeys = FIDO2 + синхронізація через хмару (iCloud Keychain, Google Password Manager). Hardware Token — фізичний пристрій без синхронізації. Passkeys зручніший (немає ризику «втрати пристрою → втрати доступу»), але приватний ключ передається в зашифрованому вигляді між пристроями екосистеми.
10. Credential Stuffing: зловмисник бере пари email:password з витоків і пробує їх на інших сайтах. Унеможливлюється виключно унікальним паролем для кожного сервісу — парольний менеджер є єдиним реалістичним способом цього досягти.
11. NIST SP 800-63B (2017): 1) Не вимагати обов'язкову ротацію без підстав (стимулює передбачувані паттерни). 2) Не вимагати певні класи символів (стимулює P@ssw0rd). 3) Перевіряти пароль по базах витоків і блокувати відомо слабкі.
12. Zero-knowledge: сервер зберігає лише зашифрований vault; ключ виводиться з master password виключно на пристрої клієнта і ніколи не передається на сервер. Навіть при зламі сервера зловмисник не може розшифрувати vault без master password.
13. Vault зашифрований master password через KDF. Зловмисник може offline-підбирати master password — але для надійних паролів це займе роки або неможливо. Тому компрометація vault ≠ компрометація паролів для тих, хто мав надійний master password.
14. OAuth 2.0 authorizes (надає access token для доступу до ресурсів), але access token не несе інформації про особу. «Увійти через Google» через чистий OAuth — помилка дизайну: ти знаєш, що клієнт авторизований, але не знаєш хто це. OIDC додає ID Token з claims про особу.
15. XSW (XML Signature Wrapping): підпис верифікується для одного XML-елемента, але застосунок обробляє інший елемент (що зловмисник додав). SAML assertion вразливий, бо XML дозволяє складні структури з вкладеними елементами.
16. `state` — випадкове значення, що клієнт надсилає при ініціації OAuth і перевіряє у callback. Захист від CSRF: зловмисник не може ініціювати OAuth flow від імені жертви, бо не знає `state` жертви.
17. В ID Token ОБОВ'ЯЗКОВО: `iss` (збігається з очікуваним IdP), `aud` (збігається з вашим client_id), `exp` (токен не прострочений), `nonce` (якщо використовувався). Без перевірки `aud` — токен для одного клієнта може бути використаний іншим.
18. PKCE (Proof Key for Code Exchange): для публічних клієнтів (SPA, мобільні) неможливо безпечно зберегти client_secret. PKCE: клієнт генерує code_verifier, хешує → code_challenge, надсилає при авторизації. При обміні на токен підтверджує verifier. Захищає від перехоплення authorization code.
19. Проблема Trojan Horse для DAC: шкідливий код запущений від імені Alice — він має всі права Alice (читати, копіювати файли). DAC не розрізняє Alice і програму від імені Alice. MAC вирішує: процес отримує мітку (тип), і навіть якщо виконується від Alice — він обмежений своєю міткою (наприклад, `httpd_t` не може читати `shadow_t`).
20. SoD (Separation of Duties) фінансовий приклад: один співробітник створює платіжне доручення, інший — підтверджує. Той хто замовляє товар ≠ той хто приймає товар ≠ той хто підтверджує оплату. Без SoD — один нечесний співробітник може ініціювати і авторизувати шахрайський платіж самостійно.
21. RBAC не враховує контекст: Alice з роллю «analyst» завжди має однаковий доступ незалежно від часу, місця, типу запиту. ABAC може: «Alice може читати фінансові дані лише в робочий час, з офісної мережі, для документів власного відділу». RBAC для цього потребував би сотень ролей.
22. ReBAC (Relationship-Based Access Control): права залежать від стосунків між суб'єктом і об'єктом. Найвідоміший приклад — Google Zanzibar (основа Google Drive): «Alice може редагувати документ, бо вона є його співавтором»; «Bob може читати, бо Alice поділилась з ним».
23. Standing Privileges: адміністратор завжди має привілейований доступ → постійно є привабливою ціллю. JIT: права видаються лише при конкретному запиті на обмежений час і автоматично відкликаються → вікно можливостей для зловмисника зводиться до мінімуму.
24. LAPS: автоматично генерує унікальний пароль локального адміністратора для кожної машини. Без LAPS: один скомпрометований ПК → lateral movement на всі 500 (однаковий пароль). З LAPS: компрометація ПК1 не дає доступу до ПК2 (різні паролі).
25. gMSA (Group Managed Service Account): Active Directory автоматично управляє паролем — він ніколи не доступний адміністраторам у відкритому вигляді, автоматично ротується, не може бути захардкоджений у код. Стандартний сервісний акаунт: пароль встановлений вручну, рідко змінюється, часто захардкоджений.
26. Session Recording: повний відеозапис привілейованої сесії адміністратора. Значення: якщо зловмисний адміністратор або атакуючий видалив системні логи — відеозапис сесії збережений в окремій системі. Дозволяє встановити точну послідовність дій при розслідуванні.
27. Conditional Access: рішення про доступ приймається динамічно на основі множини факторів (ризик сесії, стан пристрою, локація, роль). Звичайна MFA: бінарна — є MFA чи немає. Conditional Access може вимагати MFA лише при вході з нетипового місця, або блокувати доступ при high-risk сесії.
28. IAM Roles: тимчасові credentials (отримуються автоматично через IMDS), автоматично ротуються, не можуть бути «захардкоджені». IAM Users: статичні access keys — якщо втратили або витікли, можуть використовуватись роками. Для сервісів на EC2/ECS — roles є і зручніші, і безпечніші.
29. Consent Phishing: жертва натискає «Дозволити» для шкідливого OAuth-застосунку → зловмисник отримує тривалий access token без знання пароля або MFA-коду. MFA не допомагає, бо автентифікація пройшла легітимно — лише OAuth consent авторизував надмірний доступ.
30. SCIM: при деактивації акаунта в IdP (наприклад, Azure AD після звільнення) SCIM автоматично надсилає PATCH/DELETE запити до всіх підключених застосунків (Salesforce, GitHub, Slack) для деактивації акаунта. Без SCIM — ручний процес, що часто «забувається» для частини систем.
31. BeyondCorp (Google): скасовано «довірену» корпоративну мережу; кожен запит до ресурсів перевіряється на основі ідентичності + стану пристрою + контексту, незалежно від IP/мережі. VPN «розтягує» периметр до всіх домашніх роутерів; BeyondCorp усуває концепцію периметру взагалі.
32. Мікросегментація: мережа поділена на мінімальні ізольовані сегменти (до рівня workload/контейнера) з явним дозволом трафіку між ними. Ускладнює lateral movement: компрометація одного сегменту не дає автоматичного доступу до інших.
33. VPN: після підключення — доступ до всієї мережі (або великого сегменту). ZTNA: доступ лише до конкретного застосунку/ресурсу, що запитувався. ZTNA реалізує least privilege на рівні мережевого доступу.
34. UEBA: будує профіль нормальної поведінки кожного user/entity. Ризик-скоринг: при аномальній поведінці (незвичне місце, незвичний час, незвичний обсяг даних) ризик-score підвищується → система автоматично вимагає додаткову верифікацію або блокує → Continuous Verification: автентифікація не лише при вході, а постійно протягом сесії.
35. Credential Stuffing: пари логін/пароль з витоків → захист: унікальні паролі (парольний менеджер), MFA. Password Spraying: один пароль проти тисяч акаунтів → захист: блокування популярних паролів, MFA, anomaly detection (багато різних users, один пароль, короткий час).
36. Pass-the-Hash: протокол NTLM автентифікує за NT-хешем безпосередньо, без розшифрування → знання хешу = можливість автентифікації. Credential Guard: ізолює LSASS процес у VBS (Virtualization-Based Security), де Mimikatz не може дістати хеші.
37. Kerberoasting: будь-який доменний користувач може отримати Service Ticket для SPN; ticket зашифрований хешем пароля сервісного акаунту → офлайн-підбір. Найефективніший захист: gMSA (пароль керується AD, дуже довгий і складний, ніколи не доступний людям) або складний випадковий пароль 25+ символів для сервісних акаунтів.
38. Golden Ticket: зловмисник має KRBTGT-хеш → може генерувати TGT для будь-якого user на 10 років → навіть після «очищення» — ticket залишається. Необхідно двічі змінити KRBTGT пароль (AD зберігає попередній для плавної ротації), аудитувати всі DC, шукати аномальні Event ID 4769.
39. Real-time phishing (AITM): Evilginx, Modlishka — перехоплюють TOTP-код миттєво і використовують в реальному часі → проти TOTP ефективно. FIDO2: підпис прив'язаний до домену (RPID) → навіть перехоплений запит від фішингового сайту містить wrong RPID → сервер відхиляє → від FIDO2 не ефективно.

</details>
