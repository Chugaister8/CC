# Глосарій модуля 05

Усі ключові терміни модуля «Автентифікація та контроль доступу» в одному місці, за алфавітом. Українські терміни — за українською абеткою; англійські абревіатури та назви — в окремому блоці A-Z. У дужках — розділ, де термін розкрито детальніше.

## А

**Авторизація (Authorization)** — визначення того, що аутентифікованому суб'єкту дозволено робити; реалізується через моделі DAC, MAC, RBAC, ABAC. *(5.1, 5.5)*

**Автентифікація (Authentication)** — підтвердження особи суб'єкта (доведення «я — це дійсно я»); перший рубіж контролю доступу. *(5.1, 5.2)*

**Аудит (Auditing / Accounting)** — журналювання дій суб'єктів для подальшого аналізу і розслідування; четвертий елемент AAAA-фреймворку. *(5.1)*

## В

**Велике привілейоване ескалування (Privilege Escalation)** — отримання більших прав, ніж початково надано; горизонтальне (інший user з тими самими правами) або вертикальне (вищий рівень привілеїв). *(5.9)*

## Г

**Групова керована сервісна обліковий запис (gMSA)** → Group Managed Service Account. *(5.6)*

## І

**Ідентифікація (Identification)** — твердження про особу без підтвердження («я — user@example.com»); лише заявка. *(5.1)*

**Ідентичність (Identity)** — сукупність атрибутів і credentials, що однозначно описують суб'єкта в системі. *(5.1)*

## К

**Контроль доступу (Access Control)** — механізм визначення, які суб'єкти можуть виконувати які дії з якими ресурсами. *(5.5)*

**Credential Stuffing** → використання пар логін/пароль з витоків для атаки на інші сервіси; ефективний проти повторно використаних паролів. *(5.3, 5.9)*

**Creeping Privileges** → «повзучі привілеї»: накопичення зайвих прав при зміні ролей без відкликання попередніх. *(5.1)*

## М

**МФА (Multi-Factor Authentication)** — автентифікація двома або більше чинниками різних категорій (знання, володіння, ознака). *(5.2)*

**Мікросегментація (Micro-segmentation)** — поділ мережі на мінімальні ізольовані сегменти з явним дозволом трафіку; обмежує lateral movement. *(5.8)*

## Н

**Найменші привілеї (Least Privilege)** — суб'єкт отримує лише ті права, що мінімально необхідні для виконання конкретного завдання. *(5.1, 5.5)*

## О

**Облік / Облікування (Accounting)** — реєстрація дій суб'єктів для аудиту; разом з Автентифікацією і Авторизацією утворює AAA-фреймворк. *(5.1)*

## П

**Парольна фраза (Passphrase)** — довгий пароль з кількох слів; поєднує безпеку і запам'ятовуваність. *(5.3)*

**Привілейований доступ (Privileged Access)** — доступ з підвищеними правами (адміністратори, root, DBA); потребує окремих процесів захисту. *(5.6)*

## Р

**Роль (Role)** — набір прав, що призначається користувачам; основа RBAC-моделі. *(5.5)*

**Розділення обов'язків (Separation of Duties, SoD)** — принцип, що критичні дії вимагають участі кількох незалежних осіб; захист від зловживань. *(5.5)*

## С

**Сесійний токен (Session Token)** — тимчасовий ідентифікатор сесії після автентифікації; компрометація токена = компрометація сесії. *(5.4)*

**Суб'єкт (Subject)** — активний учасник, що здійснює доступ (користувач, процес, сервіс). *(5.5)*

## Ф

**Федерація (Federation)** — угода між різними доменами про взаємне довіряння результатам автентифікації. *(5.4, 5.7)*

**Фактор автентифікації (Authentication Factor)** — елемент доказу особи: щось знаєш, маєш, є, де є, що робиш. *(5.2)*

## A-Z (англійські абревіатури та назви)

**ABAC (Attribute-Based Access Control)** — модель контролю доступу, де рішення приймається на основі атрибутів суб'єкта, об'єкта, дії і середовища. *(5.5)*

**AITM (Adversary-in-the-Middle)** — атака типу MITM для перехоплення сесійних токенів після MFA в реальному часі (Evilginx, Modlishka). *(5.9)*

**AWS IAM** — Identity and Access Management для хмарної платформи Amazon Web Services; ролі, users, policies, permission boundaries. *(5.7)*

**BeyondCorp** — архітектура Zero Trust від Google, що скасовує концепцію «довіреної внутрішньої мережі». *(5.8)*

**CASB (Cloud Access Security Broker)** — інструмент видимості і контролю SaaS-застосунків у SASE-архітектурі. *(5.8)*

**Conditional Access** — механізм Microsoft Entra ID для динамічного прийняття рішень про доступ на основі ризику, пристрою, локації. *(5.7)*

**Consent Phishing** — OAuth-фішинг: жертва авторизує шкідливий OAuth-застосунок, що отримує тривалий access token. *(5.7, 5.9)*

**Credential Guard** — Windows-механізм (VBS) для ізоляції LSASS від доступу Mimikatz і PtH-атак. *(5.9)*

**DAC (Discretionary Access Control)** — модель, де власник ресурсу сам визначає права доступу; стандарт Linux/NTFS. *(5.5)*

**Entra ID** (Azure Active Directory) — хмарна IAM-платформа Microsoft; SSO, MFA, Conditional Access, PIM. *(5.7)*

**FIDO2** — стандарт фішинг-стійкої автентифікації (Fast IDentity Online 2); включає WebAuthn і CTAP2. *(5.2)*

**gMSA (Group Managed Service Account)** — Windows-сервісний акаунт з автоматичним управлінням паролем через AD; захист від Kerberoasting. *(5.6)*

**Golden Ticket** — Kerberos-атака, що дозволяє генерувати підроблені TGT для будь-якого user домену через компрометацію KRBTGT-хешу. *(5.9)*

**HOTP / TOTP** — Hash-based / Time-based One-Time Password; стандарти одноразових паролів (RFC 4226 / RFC 6238). *(5.2)*

**IAM (Identity and Access Management)** — система управління ідентичностями і правами доступу. *(5.1)*

**IdP (Identity Provider)** — система, що зберігає і верифікує ідентичності (Azure AD, Okta, Google). *(5.1, 5.4, 5.7)*

**JIT (Just-in-Time) Access** — привілейований доступ видається лише на конкретний час і для конкретного завдання; автоматично відкликається. *(5.6)*

**JML (Joiners, Movers, Leavers)** — модель управління життєвим циклом ідентичності в організації. *(5.1)*

**JWT (JSON Web Token)** — компактний формат токена для передачі claims між сторонами; структура: Header.Payload.Signature. *(5.4)*

**Kerberoasting** — атака на SPN-акаунти в Active Directory: запит Service Tickets і офлайн-підбір пароля сервісного акаунту. *(5.9)*

**KRBTGT** — спеціальний акаунт Kerberos; його компрометація призводить до Golden Ticket атаки. *(5.9)*

**LAPS (Local Administrator Password Solution)** — інструмент Microsoft для автоматичної генерації унікальних паролів локального адміністратора. *(5.6)*

**MAC (Mandatory Access Control)** — модель, де система примусово накладає мітки безпеки; AppArmor, SELinux; захист від Trojan Horse. *(5.5)*

**MFA Fatigue** — атака нескінченними push-сповіщеннями MFA до випадкового підтвердження жертвою. *(5.2)*

**OIDC (OpenID Connect)** — протокол автентифікації поверх OAuth 2.0; додає ID Token з claims про особу. *(5.4)*

**PAM (Privileged Access Management)** — технології і процеси управління привілейованим доступом: vaulting, session recording, JIT. *(5.6)*

**Passkeys** — FIDO2 credentials з синхронізацією через хмару (Apple/Google/Microsoft); фішинг-стійкий замінник паролів. *(5.2)*

**Password Spraying** — атака: один-два простих паролі проти тисяч акаунтів; уникає блокування за рахунок розподілу. *(5.9)*

**Pass-the-Hash (PtH)** — атака в Windows: використання NT-хешу для NTLM-автентифікації без знання пароля. *(5.9)*

**Pass-the-Ticket (PtT)** — атака в Kerberos: використання вкраденого TGT або Service Ticket. *(5.9)*

**PIM (Privileged Identity Management)** — Microsoft компонент для JIT-доступу до privileged roles в Azure AD. *(5.6, 5.7)*

**PKCE (Proof Key for Code Exchange)** — розширення OAuth 2.0 для публічних клієнтів; захист від перехоплення authorization code. *(5.4)*

**RBAC (Role-Based Access Control)** — права призначаються ролям, користувачі отримують права через членство в ролях. *(5.5)*

**ReBAC (Relationship-Based Access Control)** — права залежать від стосунків між суб'єктом і об'єктом; реалізація: Google Zanzibar. *(5.5)*

**RPID (Relying Party ID)** — домен, що прив'язується до FIDO2 credentials; захищає від AITM-фішингу. *(5.2)*

**SAML 2.0** — XML-стандарт для передачі автентифікаційних assertions між IdP і SP; використовується в enterprise B2B SSO. *(5.4)*

**SASE (Secure Access Service Edge)** — архітектурна концепція, що об'єднує мережеві функції (SD-WAN) і безпеку (CASB, SWG, ZTNA) у хмарній платформі. *(5.8)*

**SCIM 2.0** — REST API стандарт для автоматичного провізіонингу і деprovisioning між IdP і застосунками. *(5.1, 5.7)*

**Silver Ticket** — Kerberos-атака через компрометацію хешу конкретного сервісного акаунту (не KRBTGT). *(5.9)*

**SoD (Separation of Duties)** — жодна особа не повинна мати повноваження для самостійного виконання критичної дії від початку до кінця. *(5.5)*

**SP (Service Provider) / RP (Relying Party)** — система, що надає сервіс і довіряє IdP для автентифікації. *(5.1, 5.4)*

**SSO (Single Sign-On)** — механізм одноразової автентифікації з доступом до множини сервісів. *(5.4)*

**Standing Privileges** — постійно наявні привілейовані права; протилежність JIT. *(5.6)*

**TOTP (Time-based One-Time Password)** — одноразовий код, що змінюється кожні 30 секунд (RFC 6238); вразливий до AITM-фішингу. *(5.2)*

**UEBA (User and Entity Behavior Analytics)** — система поведінкового аналізу для виявлення аномалій і ризиків. *(5.8)*

**WebAuthn** — Web API стандарт W3C для FIDO2-автентифікації в браузерах. *(5.2)*

**XSW (XML Signature Wrapping)** — атака на SAML: підпис верифікується для одного XML-елемента, але застосунок обробляє інший. *(5.4)*

**ZTNA (Zero Trust Network Access)** — заміна VPN, що надає доступ лише до конкретного застосунку (не до всієї мережі). *(5.8)*
