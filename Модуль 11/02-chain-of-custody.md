# 11.2. Chain of Custody і правові аспекти

Бездоганний технічний аналіз марний, якщо не можна довести, що зібрані докази не були змінені між моментом збору і моментом представлення висновків. Chain of Custody — документований ланцюжок, що відповідає на питання «хто, коли і що робив з цим доказом» від моменту виявлення до фінального звіту. Прогалина в цьому ланцюжку — навіть одна незадокументована година доступу до диска — може поставити під сумнів усе розслідування, незалежно від того, наскільки технічно бездоганним був сам аналіз.

> 📖 Ключові терміни — у [глосарії модуля](00-glosariy.md).

## Chain of Custody: що це і навіщо

**Chain of Custody (ланцюжок зберігання доказів)** — хронологічна документація, що фіксує послідовність контролю, передачі, аналізу і розпорядження фізичними або цифровими доказами.

**Мінімальний набір інформації для кожного доказу:**

```
Для кожного предмету доказів:
├── Унікальний ідентифікатор (evidence tag/number)
├── Опис (тип носія, серійний номер, модель)
├── Дата і час вилучення/збору
├── Місце вилучення
├── Хто вилучив (ім'я, посада, контакти)
├── Хеш-сума (MD5 і/або SHA-256) на момент збору
└── Кожна подальша передача:
    ├── Від кого → до кого
    ├── Дата і час
    ├── Мета передачі
    └── Підпис обох сторін (фізичний або цифровий)
```

**Приклад форми Chain of Custody:**

```
EVIDENCE CHAIN OF CUSTODY FORM

Case Number: CASE-2024-0142
Evidence Item: #001
Description: Dell Latitude 5420 Laptop, S/N: ABC123XYZ
Collected From: Server Room, 3rd Floor, офіс Компанія
Collected By: Іван Петренко, Security Analyst
Date/Time Collected: 2024-01-15 14:32 EET
Hash (SHA-256) at collection: 8f3a2c91e7b4d5f6...

| # | Date/Time | Released By | Received By | Purpose | Location |
|---|-----------|-------------|--------------|---------|----------|
| 1 | 2024-01-15 14:35 | I. Petrenko | M. Kovalenko | Imaging | Lab |
| 2 | 2024-01-15 16:20 | M. Kovalenko | Evidence Locker | Storage | Locker A-12 |
| 3 | 2024-01-20 09:00 | Evidence Locker | I. Petrenko | Analysis | Lab |
```

## Hash-верифікація: математичний доказ незмінності

Криптографічний хеш — фундаментальний інструмент доведення цілісності доказів (детально криптографічна основа — модуль 04).

```bash
# При зборі доказу — обчислити хеш ОРИГІНАЛУ і КОПІЇ
sha256sum /dev/sdb > original_hash.txt
# 8f3a2c91e7b4d5f6a9c2e8b1d4f7a0c3e6b9d2f5a8c1e4b7d0f3a6c9e2b5d8f1  /dev/sdb

# Створення біт-копії з write-blocker
dd if=/dev/sdb of=evidence_001.dd bs=4M status=progress conv=noerror,sync

# Верифікація: хеш копії МАЄ збігатись з хешем оригіналу
sha256sum evidence_001.dd
# Якщо хеші НЕ збігаються — копія пошкоджена або неповна

# Найкраща практика: створення ДВОХ копій
# 1. Working copy — для аналізу (можна модифікувати при дослідженні)
# 2. Archive copy — недоторкана, лише для верифікації
```

**Чому MD5 все ще використовується у форензиці попри криптографічну застарілість:** MD5 зламаний для криптографічних цілей (можна створити колізію — модуль 04), але для форензичної верифікації цілісності (виявлення випадкового пошкодження, а не навмисної підробки) MD5 + SHA-256 разом залишаються стандартом індустрії — швидкість MD5 і надійність SHA-256 доповнюють одне одного.

## Write-Blockers: апаратний захист від модифікації

**Write Blocker** — апаратний або програмний пристрій, що дозволяє читання з носія, але фізично блокує будь-які команди запису.

```
Без write-blocker:
Дослідник підключає диск до ОС
→ ОС автоматично записує метадані (mount timestamps,
   journal updates, swap файли)
→ Оригінальні дані змінені ще до початку аналізу!

З write-blocker:
Диск підключається через write-blocker пристрій
→ Будь-яка команда запису фізично відхиляється на рівні
   контролера, незалежно від того, що "намагається" зробити ОС
→ Оригінал гарантовано незмінний
```

**Виробники апаратних write-blockers:** Tableau, WiebeTech, CRU. Програмні альтернативи (Linux): монтування з прапором `ro` (read-only) + блокування на рівні ядра — менш надійно, ніж апаратний рівень, але прийнятно для менш критичних розслідувань.

## Правовий статус цифрових доказів в Україні

```
Нормативна база для цифрових доказів в Україні:

Кримінальний процесуальний кодекс України (КПК):
  Ст. 99 — Документи як джерела доказів
  Ст. 159-166 — Тимчасовий доступ до речей і документів
  Ст. 236 — Порядок проведення обшуку

ЗУ "Про основні засади забезпечення кібербезпеки України":
  Визначає суб'єктів реагування на кіберінциденти
  CERT-UA — координація розслідування інцидентів

Закон України "Про захист персональних даних":
  Регулює обробку персональних даних при розслідуванні

Важливо: цифрові докази для кримінального провадження
мають збиратись із дотриманням процесуальних вимог КПК —
залучення слідчого, понятих, відповідне документування.
Внутрішнє корпоративне розслідування має менш формальні
вимоги, але має готуватись так, ніби може знадобитись
у суді (цивільний позов, трудовий спір, страховий case).
```

**Практична порада для корпоративних розслідувачів:** навіть якщо розслідування зараз не передбачає кримінального провадження, документуйте процес так, ніби це знадобиться в суді. Якість документації не можна «додати заднім числом» — її можна лише вести з самого початку або не мати взагалі.

## Admissibility: чому докази можуть бути відхилені судом

```
Найпоширеніші підстави для відхилення цифрових доказів:

1. Порушений Chain of Custody
   → Прогалини в документації, незрозуміло хто мав доступ

2. Незаконний збір
   → Доступ без належних повноважень або дозволу

3. Технічна некомпетентність
   → Дослідник не може пояснити методологію або інструменти

4. Відсутність верифікації цілісності
   → Немає хешів, неможливо довести що дані не змінені

5. Упереджена методологія
   → Дослідник шукав лише підтверджуючі докази (confirmation bias)
```

## Privacy і Proportionality при розслідуванні

Корпоративні розслідування (наприклад, підозра на insider threat) мають балансувати між потребою розслідування і правами працівника на приватність.

```
Принципи пропорційності для внутрішнього розслідування:

✅ Виправдано:
- Аналіз корпоративної пошти на предмет витоку даних
  (за наявності чіткої політики моніторингу)
- Перегляд логів доступу до файлів на корпоративних системах
- Аналіз корпоративного пристрою при підозрі на зловживання

⚠️ Потребує юридичної консультації:
- Доступ до особистих месенджерів на корпоративному
  пристрої (навіть якщо технічно можливо)
- Розслідування поза робочим часом активності
- BYOD пристрої — межі корпоративного доступу обмежені

❌ Зазвичай неприйнятно без окремого дозволу:
- Доступ до особистих акаунтів працівника
- Приховане відеоспостереження без повідомлення
- Розслідування, не пов'язане з законним бізнес-інтересом
```

## Документування методології: Reproducibility

ACPO Принцип 3 вимагає, щоб незалежна третя сторона могла відтворити процес і дійти того самого результату. Це означає документування:

```
Для кожного кроку аналізу:
├── Точна команда або дія, що виконувалась
├── Версія інструменту (Volatility 3.x, Autopsy 4.x тощо)
├── Час виконання
├── Хто виконував
└── Точний результат (з посиланням на файл доказу/скріншот)

Приклад запису в журналі дослідження:

[2024-01-20 10:15:32] Аналітик: I.Petrenko
Команда: volatility3 -f memory.raw windows.pslist
Версія: Volatility 3.2.0
Мета: Перелік активних процесів на момент збору пам'яті
Результат: Виявлено процес "svch0st.exe" (PID 4521) —
           підозрілий через типову typosquatting-назву
           (легітимний процес: svchost.exe)
           Збережено: output/pslist_20240120.txt
```

## Міні-вправа

1. Створіть шаблон форми Chain of Custody для вашої організації (або навчального проекту) на основі прикладу вище.
2. Розгляньте сценарій: ви знайшли підозрілий файл на робочому ноутбуці співробітника. Опишіть покроково, як ви б забезпечили Chain of Custody від моменту виявлення до передачі результатів керівництву.
3. Обчисліть SHA-256 хеш будь-якого тестового файлу на своєму комп'ютері, скопіюйте файл, перевірте чи хеш копії збігається.

```bash
# Тестова вправа
echo "Test evidence file" > evidence.txt
sha256sum evidence.txt
cp evidence.txt evidence_copy.txt
sha256sum evidence_copy.txt
# Хеші мають бути ідентичні
```
