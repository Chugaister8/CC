# 8.7. Industrial IoT і OT/SCADA Security

Якщо IoT-атака на розумний дім — це незручність, то атака на промислову систему управління — це зупинка заводу, аварія на електростанції або розлив небезпечних речовин. Операційні технології (OT) існують десятиліттями раніше за кібербезпеку і були спроектовані для надійності і безперервності, а не для захисту від зловмисників. Інтеграція промислових систем з інтернетом за останнє десятиліття створила нову категорію критичних вразливостей.

> 📖 Ключові терміни — у [глосарії модуля](00-glosariy.md).

## Ключова термінологія

**OT (Operational Technology)** — апаратне і програмне забезпечення, що контролює і моніторить фізичні процеси: обладнання, виробничі лінії, інфраструктуру.

**ICS (Industrial Control Systems)** — загальний термін для промислових систем управління.

**SCADA (Supervisory Control and Data Acquisition)** — системи диспетчерського управління і збору даних; використовуються для управління розподіленою інфраструктурою (електромережа, водопостачання, газопроводи).

**PLC (Programmable Logic Controller)** — промисловий мікроконтролер для автоматизації конкретних задач (відкриття клапана, управління двигуном).

**DCS (Distributed Control System)** — розподілена система управління для комплексних виробничих процесів (хімічні заводи, нафтопереробка).

**HMI (Human-Machine Interface)** — панель оператора для моніторингу і управління.

## Purdue Model: ієрархія промислових мереж

**Purdue Reference Model** (ISA-95) — стандартна архітектура ізоляції IT і OT:

```
Рівень 5: Enterprise Network (корпоративна мережа, ERP)
             │ ← Firewall + DMZ
Рівень 4: Business Planning (бізнес-планування, MES)
             │ ← Firewall
Рівень 3: Operations Management (управління виробництвом, Historian, SCADA)
             │ ← Industrial DMZ (Data Diode або Firewall)
Рівень 2: Process Control (Control Room, HMI, Engineering Workstation)
             │ ← Firewall або Air Gap
Рівень 1: Basic Control (PLC, DCS, RTU)
             │
Рівень 0: Physical Process (сенсори, актуатори, фізичне обладнання)
```

**Ключовий принцип:** трафік не має переходити напряму з рівня 5 до рівня 0. Кожен перехід через рівні потребує явного дозволу і контролю.

**IT/OT конвергенція** — сучасні тенденції порушують цю ізоляцію:
- Remote monitoring через інтернет.
- Cloud-based SCADA.
- Predictive maintenance через AI (потребує raw data з рівня 0–2 у хмарі).
- Vendor remote access для технічного обслуговування.

## Відмінності IT і OT безпеки

| Аспект | IT | OT |
|---|---|---|
| Пріоритет CIA | Confidentiality → Integrity → Availability | **Availability → Safety → Integrity** |
| Оновлення | Регулярні, автоматичні | Рідкісні, тестовані місяцями |
| Перезавантаження | Норма | Може означати зупинку виробництва |
| Терміни служби | 3–5 років | 20–30 років |
| Протоколи | Стандартні (TCP/IP, TLS) | Специфічні (Modbus, DNP3, PROFIBUS, OPC-UA) |
| Патчування | Щотижня | Ніколи або раз на рік під час maintenance window |
| Ізоляція | Мережеві сегменти | Air gap або Industrial DMZ |
| Antivirus | Обов'язковий | Часто неможливий (legacy ОС, реальний час) |

**Чому OT не можна патчувати звично:**
- Windows XP або Windows Server 2003 на критичних PLC/HMI.
- Виробник PLC перестав підтримувати ПЗ, але обладнання ще справне.
- Будь-яке оновлення потребує тестування на ідентичному стенді (кілька місяців).
- Зупинка для патчування = зупинка виробництва = мільйонні збитки.

## Промислові протоколи і їх вразливості

### Modbus (1979)

Найстаріший і найпоширеніший промисловий протокол. Розроблений до появи концепції кібербезпеки.

**Критичні проблеми Modbus:**
- **Відсутність автентифікації:** будь-хто в мережі може видавати команди PLC.
- **Відсутність шифрування:** всі дані у відкритому вигляді.
- **Відсутність авторизації:** немає різниці між read і write правами.
- **Немає цілісності:** немає перевірки що команда не модифікована.

```python
# Демонстрація незахищеності Modbus (лише для власного устаткування!)
# pip install pymodbus
from pymodbus.client import ModbusTcpClient

# Підключення без автентифікації
client = ModbusTcpClient('192.168.1.100', port=502)
client.connect()

# Читання регістрів — будь-хто в мережі може це зробити
result = client.read_holding_registers(0, 10, unit=1)
print(f"Регістри: {result.registers}")

# Запис команди — теж без будь-якої автентифікації!
# client.write_register(0, 1, unit=1)  # Потенційно небезпечно!
```

**Захист:** Modbus TCP Secure (рідко реалізований), ізоляція в окремому VLAN, Unidirectional Security Gateway (Data Diode).

### OPC UA (Unified Architecture)

**OPC UA** — сучасний промисловий протокол з вбудованою безпекою (TLS, X.509, автентифікація). Рекомендований для нових розробок.

## Stuxnet: перша кіберзброя для промислових систем

**Stuxnet (2010)** — перша публічно задокументована кіберзброя, що фізично пошкодила реальне обладнання.

**Ціль:** іранська програма ядерного збагачення; завод Натанз.

**Механізм атаки:**
1. Розповсюдження через USB (zero-day вразливість Windows LNK).
2. Пошук специфічних Siemens S7-315 і S7-417 PLC з певним конфігурацією.
3. Підміна коду у PLC: центрифуги для збагачення урану прискорювались і сповільнювались за межі норми.
4. Водночас — показ операторам нормальних показників (приховування атаки).

**Результат:** знищено 1 000–5 000 центрифуг; програма сповільнена на 1–2 роки.

**Уроки Stuxnet:**
- Air Gap не захищає від USB-зараження.
- OT-системи без захисту вразливі до цілеспрямованих атак.
- Кіберзброя може фізично пошкоджувати обладнання.
- Атаки на критичну інфраструктуру є реальністю (не теорією).

## ICS/SCADA в контексті українського збройного конфлікту

Атаки на критичну інфраструктуру України є частиною гібридної війни Росії:

**Industroyer / Crashoverride (2016)** — атака на українську енергомережу, що залишила частину Києва без електроенергії. Перша кіберзброя, спрямована безпосередньо на промислові системи управління електромережею.

**Industroyer2 (2022, CERT-UA)** — нова версія кіберзброї, виявлена CERT-UA у квітні 2022. Атака на підстанцію в Україні була зупинена до виконання.

**Sandworm / APT44** — російська APT-група, що спеціалізується на атаках на критичну інфраструктуру; відповідальна за NotPetya, Industroyer і Industroyer2.

## Захист OT/SCADA середовища

**Принципи Defense-in-Depth для OT:**

1. **Мережева ізоляція:** суворе дотримання Purdue Model; Industrial DMZ між IT і OT.
2. **Whitelist for network traffic:** у OT-мережі дозволяти лише відомий легітимний трафік (Modbus від конкретних HMI до конкретних PLC).
3. **OT-специфічний моніторинг:** Dragos, Claroty, Nozomi Networks — системи виявлення аномалій, що розуміють промислові протоколи.
4. **Фізичний захист:** обмеження фізичного доступу до PLC і HMI.
5. **USB-контроль:** забороняти або контролювати USB-пристрої в OT-мережах.
6. **Vendor Remote Access:** тільки через Jump Server або bastion host; час обмежений; записується.
7. **Резервне живлення і backup конфігурацій:** регулярне резервне копіювання конфігурацій PLC/HMI поза основною мережею.

## Міні-вправа

Знайдіть у публічних джерелах (CISA, CERT-UA, ICS-CERT) будь-який звіт про атаку на критичну інфраструктуру за останній рік:

1. Яку систему атаковано (електромережа, водопостачання, виробництво)?
2. Яким вектором здійснено початковий доступ?
3. Які TTP (за MITRE ATT&CK for ICS) застосовував зловмисник?
4. Які контрзаходи рекомендують авторитети?
