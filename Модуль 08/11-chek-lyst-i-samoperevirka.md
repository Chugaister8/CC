# 8.11. Чек-лист і самоперевірка

## Підсумок модуля: ключові ідеї

1. **iOS і Android мають різні моделі безпеки.** iOS — закрита екосистема з Secure Enclave; Android — відкрита з SELinux і Keystore. Жодна не «безпечніша» абсолютно — різні профілі ризику.
2. **Мобільне шкідливе ПЗ — реальна загроза.** Banking Trojans для Android, Pegasus для iOS; SIM swap і SS7 роблять SMS-2FA ненадійним.
3. **OWASP Mobile Top 10 — чек-лист для розробника.** M1 (credential storage), M3 (auth), M5 (insecure comms), M9 (data storage) — найчастіші джерела вразливостей.
4. **MDM/MAM — необхідність для корпоративних пристроїв.** BYOD без MAM означає некероване зберігання корпоративних даних; Work Profile ізолює без порушення приватності.
5. **IoT — найбільша незахищена поверхня атаки.** Дефолтні паролі, відсутність оновлень, незашифрований Modbus — типовий IoT-пристрій 2024 року.
6. **VLAN-сегментація IoT — мінімальний необхідний захист.** Ізоляція IoT від корпоративної мережі зупиняє lateral movement навіть при компрометації пристрою.
7. **OT/SCADA — критична інфраструктура з унікальними вимогами.** Пріоритет: Availability → Safety; патчування обмежене; промислові протоколи не мають вбудованої автентифікації.
8. **Stuxnet і Industroyer — кіберзброя проти реального обладнання.** Атаки на SCADA можуть фізично пошкоджувати інфраструктуру — для України це не теорія.

## Практичний чек-лист

### Особистий мобільний пристрій

| Захід | iOS | Android |
|---|---|---|
| PIN/пароль ≥ 6 символів | Face/Touch ID + буквено-цифровий код | PIN ≥ 6 або пароль |
| Блокування після 1 хвилини | Автоблокування → 1 хвилина | Дисплей → Сон → 1 хвилина |
| Захист від несанкціонованого доступу | Стерти після 10 спроб: ON | Factory reset після N спроб |
| Оновлення ОС | Налаштування → Загальні → Оновлення | Налаштування → Безпека → Security update |
| Геолокація | «Під час використання» для більшості | «Лише під час використання» |
| Захист мікрофона | Конфіденційність → Мікрофон → перегляд | Менеджер дозволів → Мікрофон |
| VPN у публічних мережах | VPN профіль налаштовано | Always-on VPN або ручне вмикання |

### Корпоративна мережа з IoT

| Захід | Пріоритет | Перевірка |
|---|---|---|
| IoT у окремому VLAN | 🔴 Must | `ip addr show` на IoT пристрої — не 192.168.10.x |
| Firewall блокує IoT→Corporate | 🔴 Must | nmap з IoT VLAN на Corporate — порти закриті |
| Дефолтні паролі змінено | 🔴 Must | Спроба admin/admin на кожному пристрої |
| Telnet (23) вимкнено | 🔴 Must | nmap -p 23 IoT_network — закрито |
| MQTT з TLS (8883) | 🟠 Should | Port 1883 закрито або з auth |
| Firmware актуальна | 🟠 Should | Перевірка версій проти NVD |
| Інвентаризація IoT пристроїв | 🟠 Should | Реєстр всіх підключених пристроїв |
| Моніторинг аномалій | 🟡 Advanced | SIEM з IoT правилами |

## Орієнтовні відповіді

<details>
<summary>Розгорнути відповіді</summary>

1. Secure Enclave — окремий процесор у Apple Silicon з власною ОС. Захищає: ключі шифрування диска (AES-256), біометричні шаблони Face ID/Touch ID, Apple Pay credentials. Навіть при компрометації iOS — Secure Enclave ізольований.
2. SELinux реалізує Mandatory Access Control (MAC) — обов'язкові мітки безпеки для кожного файлу, процесу і сокету. Навіть root-процес обмежений SELinux правилами, на відміну від стандартного DAC (chmod), де root має повний доступ.
3. Verified Boot перевіряє цілісність кожного компонента завантаження через Merkle Tree хешів. Захищає від модифікації системних розділів (завантаження кастомного ядра, TWRP recovery без попередження). Якщо зміна виявлена — пристрій не завантажується або показує попередження.
4. Normal: надаються автоматично без запиту (інтернет, часовий пояс). Dangerous: запитуються у runtime (камера, мікрофон, геолокація, SMS, контакти). Signature: лише для застосунків того ж розробника. Ключове — Dangerous дозволи дають доступ до чутливих даних і вимагають явної згоди.
5. Jailbreak порушує Code Signing (будь-який код виконується без підпису Apple) і Sandbox (застосунки можуть читати дані інших). Шкідливий застосунок поза App Store може встановитись без обмежень. Також безпечні платіжні застосунки блокуються.
6. Evil Twin: зловмисник розгортає AP з тим самим SSID; пристрій підключається автоматично; весь трафік через зловмисника. VPN шифрує трафік до виходу з пристрою — навіть через Evil Twin зловмисник бачить лише зашифрований тунель.
7. SS7 (1975) — протокол маршрутизації дзвінків без автентифікації. Зловмисник з SS7-доступом може перехопити будь-яке SMS. Надійніша альтернатива: TOTP-застосунок (Google Authenticator) або FIDO2 — не залежать від телефонної мережі.
8. SIM Swap: зловмисник переконує оператора перенести номер на нову SIM. Результат: всі SMS (OTP, банківські коди, відновлення паролів) надходять зловмиснику. Захист: PIN на SIM/акаунті у оператора; відмова від SMS 2FA.
9. Juice Jacking: шкідливий USB-зарядний пристрій (публічний порт) одночасно передає дані. Захист: USB Data Blocker (пропускає лише живлення, блокує дата-лінії); зарядка через власний адаптер до розетки (не USB-порту).
10. SharedPreferences — звичайний XML файл у `/data/data/app/shared_prefs/`. При root-доступі або через adb backup — читається без шифрування. Правильно: EncryptedSharedPreferences або Android Keystore. UserDefaults iOS аналогічно — незахищений файл у app container.
11. Certificate Pinning: застосунок перевіряє що сервер має конкретний сертифікат (або публічний ключ), а не будь-який виданий довіреним CA. Запобігає MITM через підроблені сертифікати. Ризик: при ротації сертифіката сервера — старі версії застосунку перестають працювати.
12. Mass Assignment: API автоматично мапить параметри запиту на поля об'єкта. Зловмисник може передати `{"role": "admin"}` у тілі запиту і змінити власну роль. Захист: явний whitelist дозволених для оновлення полів.
13. Overlay Attack (M9/Accessibility) або Banking Trojan: шкідливий застосунок відображає підроблений екран поверх банківського і перехоплює введені дані включаючи OTP. Наприклад, Sharkbot і Cerberus використовують accessibility service для overlay.
14. MDM: управляє всім пристроєм (конфігурація ОС, всі застосунки, Remote Wipe всіх даних). MAM: управляє лише корпоративними застосунками і їх даними (selective wipe). При BYOD: MAM без MDM — організація не стирає особисті фото.
15. Android Work Profile: окремий контейнер на пристрої для корпоративних застосунків. IT-адміністратор керує Work Profile, не має доступу до особистих даних і застосунків. При звільненні — видаляється лише Work Profile.
16. Zero-Touch Enrollment (Apple DEP): новий пристрій при першому увімкненні автоматично реєструється в MDM організації без участі IT. Конфігурується через Apple Business Manager — пристрій «знає» куди підключитись ще з заводу.
17. MTD (Mobile Threat Defense) аналізує загрози в реальному часі: підозрілі Wi-Fi мережі (Evil Twin), шкідливі застосунки, jailbreak/root, вразливості ОС. Інтегрується з MDM для enforcement: пристрій з виявленою загрозою отримує BLOCKED статус.
18. Рівень 1 (Perception): сенсори, актуатори, MCU (що вимірює і виконує). Рівень 2 (Network): протоколи передачі (Wi-Fi, Zigbee, MQTT), Gateway. Рівень 3 (Application): хмарна платформа, аналітика, управління парком пристроїв.
19. MQTT — pub/sub протокол на базі TCP для IoT. Легший за HTTP (менший header), підтримує QoS, broker-based (пристрій публікує в broker, інший підписується). Для ресурсообмежених пристроїв з повільними каналами — значно ефективніший.
20. Незахищений MQTT: анонімний доступ без автентифікації (будь-хто підключається); відсутність ACL (будь-хто публікує в будь-який topic); wildcard підписка # — видно весь трафік; відсутній TLS — перехоплення в мережі.
21. Mirai сканував інтернет на відкритий Telnet (23) і підбирав 61 дефолтну пару логін/пароль для IP-камер, DVR, маршрутизаторів. 600 000+ зомбі-пристроїв отримали команду flood → DDoS 1.2 Tbps на Dyn DNS → пів-інтернету США недоступно.
22. IoT виробники часто зупиняють підтримку через 2–3 роки. Пристрій може виконувати функцію ще 10 років — з невиправленими CVE. Оновлення складне (відсутній механізм, страх зламати функціональність). Наслідок: тисячі вразливих пристроїв у мережах.
23. Secure Boot перевіряє підпис кожного компонента завантаження (bootloader → ядро → застосунок). Якщо зловмисник підмінив прошивку — підпис не збігається → завантаження блоковане. Ланцюжок довіри починається з незмінного Boot ROM.
24. Secure Element — окремий захищений мікросхема для криптографічних операцій. Ключі не можуть бути витягнуті навіть при повній компрометації головного процесора. Захищає: автентифікаційні ключі, платіжні credentials, сертифікати пристрою.
25. Crypto Erase: замість фізичного перезапису кожного сектора — знищуємо ключ шифрування. Всі дані залишаються на носії, але стають нечитабельними (зашифровані без ключа). Швидко (секунди) і гарантовано для SSD/flash де фізичне перезаписування ненадійне.
26. В OT зупинка (низька Availability) може означати аварію, зупинку виробництва, небезпеку для людей. Конфіденційність важлива, але оператор завжди повинен мати доступ до контрольних панелей. Safety (безпека фізичних процесів) — найвищий пріоритет.
27. Purdue Model ієрархічно розділяє IT і OT: Enterprise (рівень 5) → Process Control (рівень 2) → Physical (рівень 0). Трафік між рівнями проходить через firewall або Industrial DMZ. Атака з IT не досягає PLC рівня 1 без перетину контрольованих кордонів.
28. Modbus (1979): відсутня автентифікація (будь-хто в мережі посилає команди PLC), відсутнє шифрування (всі дані у відкритому вигляді), відсутня авторизація (немає різниці між read/write правами). Розроблений до концепції кібербезпеки.
29. Stuxnet: розповсюдження через USB (не через мережу); WinCC/Step 7 software пошук; підміна коду PLC Siemens S7; центрифуги прискорювались/сповільнювались понад норму; оператори бачили нормальні показники (маскування). Air Gap обійдено через USB.
30. Industroyer (2016) і Industroyer2 (2022): перша кіберзброя для атаки на електромережу. Безпосередньо управляла промисловими протоколами (IEC 101, 104, 61850). 2016: частина Києва без електрики. 2022: атака зупинена CERT-UA до виконання. Демонструє реальну загрозу для критичної інфраструктури України.
31. Біометрія (Touch/Face ID) — зручна, але не захищає від примусового розблокування. У деяких країнах правоохоронці можуть примусити притулити палець. Пін/пароль — знання, яке не можна фізично «взяти». Рекомендація при перетині ризикових кордонів: вимкнути biometric, залишити PIN.
32. Work Profile: MDM бачить корпоративні застосунки в Work Profile (compliance, версії, налаштування). MDM НЕ бачить: особисті застосунки, особисті фото, особисті контакти, особисте листування. При Selective Wipe — видаляється лише Work Profile.
33. PPTP — повністю зламаний (MS-CHAP v2 ключ підбирається за хвилини). L2TP без IPSec — незашифрований. Рекомендовані: WireGuard (сучасний, швидкий), IKEv2/IPSec (вбудований у iOS/Android, надійний), OpenVPN (перевірений часом).
34. USB Data Blocker (PortaPow, eSynce) — фізичний адаптер, що пропускає лише +5V (живлення) і блокує D+ і D- лінії (дані). Захищає від Juice Jacking: публічний USB-порт може передавати дані на пристрій, встановлювати шкідливі драйвери або зчитувати файли.
35. IoT VLAN: IoT пристрої в окремій підмережі (VLAN 20) ізольовані від корпоративної (VLAN 10). Firewall блокує IoT → Corporate трафік. Якщо IoT-камера скомпрометована — зловмисник у VLAN 20, не може напряму атакувати VLAN 10 (ERP, файлові сервери).
36. NAC Device Fingerprinting: аналізує MAC OUI, DHCP options, Bonjour/mDNS, User-Agent і класифікує пристрій автоматично. IP-камера (за OUI Hikvision і DHCP vendor string) → VLAN 20 IoT автоматично. Ноутбук з 802.1x автентифікацією → VLAN 10 Corporate.
37. Аномалії IoT: IP-камера підключається до зовнішнього IP о 3:00 (може бути botnet C2); сенсор намагається SSH підключення (lateral movement); різке зростання DNS-запитів (ексфільтрація або DGA malware); MQTT публікація з нового IP (скомпрометований пристрій або зловмисник).
38. Деcommissioning IoT: видалити з IoT платформи (дереєстрація); відкликати X.509 сертифікат пристрою; Factory Reset (очищення Wi-Fi паролів, API keys, конфігурацій); відключити від хмарного акаунту; задокументувати; фізична утилізація згідно регламенту.
39. Підозрілі дозволи для flashlight (ліхтарика): `READ_SMS` (читання SMS — навіщо ліхтарику?) і `ACCESS_FINE_LOCATION` (точна геолокація — теж не потрібна). Також підозрілі: RECORD_AUDIO, READ_CONTACTS, GET_ACCOUNTS.
40. Overlay attack: `android.permission.SYSTEM_ALERT_WINDOW` (відображення поверх інших вікон) або `android.permission.BIND_ACCESSIBILITY_SERVICE` (accessibility — може читати і симулювати весь UI). Ці дозволи в застосунку «flashlight» або «calculator» — явна ознака підозрілого функціоналу.

</details>
