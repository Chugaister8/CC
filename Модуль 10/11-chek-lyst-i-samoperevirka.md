# 10.11. Чек-лист і самоперевірка

## Підсумок модуля: ключові ідеї

1. **Firewall еволюціонував від простого фільтра до контекстної системи.** Кожне покоління — пакетна фільтрація, stateful inspection, application layer, NGFW — з'явилось як відповідь на конкретний клас атак, що попереднє покоління не могло зупинити.
2. **IDS виявляє, IPS блокує.** Сигнатурне виявлення точне для відомих загроз; аномальне — здатне виявляти нове, але з вищим рівнем false positives.
3. **IPSec потужний, але складний; WireGuard — сучасна альтернатива з мінімальною кодовою базою.** Вибір протоколу VPN — компроміс між сумісністю, продуктивністю і простотою аудиту.
4. **Сегментація — найефективніший інструмент обмеження поширення атаки.** Від VLAN до мікросегментації до ZTNA — еволюція в бік дедалі дрібнішої і динамічнішої ізоляції.
5. **NAC контролює «хто і що» підключається до мережі ще до надання доступу.** 802.1X + RADIUS + Posture Assessment — стандартна тріада корпоративного NAC.
6. **DNS — одночасно інструмент атаки і захисту.** DNSSEC захищає від підробки відповідей; DNS Filtering блокує шкідливі домени; DNS Tunneling — прихований канал для зловмисників.
7. **Proxy і SWG контролюють вихідний трафік користувачів.** SSL Inspection — потужний, але етично і технічно складний компроміс приватності.
8. **WPA3 усуває фундаментальні вразливості WPA2** через SAE замість вразливого 4-way handshake, стійкий до offline-атак на пароль.
9. **Мережевий моніторинг без baseline — моніторинг без сенсу.** NetFlow для метаданих у масштабі, packet capture для глибокого аналізу конкретних інцидентів.

## Практичний чек-лист

### Периметр і сегментація

| Захід | Перевірка | Пріоритет |
|---|---|---|
| Default Deny на firewall | Перевірка останнього правила в ruleset | 🔴 Must |
| Stateful inspection увімкнено | Конфігурація firewall | 🔴 Must |
| NGFW з App-ID для критичних сегментів | Наявність DPI | 🟠 Should |
| Сегментація IT/OT/Guest/IoT | VLAN-схема задокументована | 🔴 Must |
| DMZ для публічних сервісів | Архітектурна діаграма | 🔴 Must |
| Мікросегментація для критичних workload | Naming policy в SG/NSG | 🟡 Advanced |

### Виявлення і запобігання

| Захід | Перевірка | Пріоритет |
|---|---|---|
| NIDS на ключових точках мережі | Розміщення сенсорів | 🟠 Should |
| Сигнатури оновлюються автоматично | Дата останнього оновлення rules | 🔴 Must |
| Threat Intelligence Feed підключено | Інтеграція з NIDS/IPS | 🟠 Should |
| Tuning false positives виконувався | Дата останнього tuning | 🟠 Should |

### Віддалений доступ

| Захід | Перевірка | Пріоритет |
|---|---|---|
| VPN з MFA | Конфігурація автентифікації | 🔴 Must |
| Split tunneling вимкнено для чутливих ролей | VPN policy | 🟠 Should |
| Застарілі протоколи (PPTP) вимкнені | Конфігурація VPN gateway | 🔴 Must |
| Always-On VPN для керованих пристроїв | MDM policy | 🟡 Advanced |

### DNS і веб-трафік

| Захід | Перевірка | Пріоритет |
|---|---|---|
| DNS Filtering увімкнено | Перевірка через тестовий шкідливий домен | 🟠 Should |
| DNSSEC валідація на резолвері | `dig +dnssec` тест | 🟡 Advanced |
| SWG для вихідного веб-трафіку | Конфігурація проксі | 🟠 Should |

### Бездротова мережа

| Захід | Перевірка | Пріоритет |
|---|---|---|
| WPA3 (або WPA2-Enterprise мінімум) | Конфігурація AP | 🔴 Must |
| Enterprise Wi-Fi з 802.1X для співробітників | RADIUS-інтеграція | 🟠 Should |
| Гостьова мережа ізольована | VLAN-перевірка | 🔴 Must |
| Rogue AP detection увімкнено | WIPS-конфігурація | 🟡 Advanced |

## Орієнтовні відповіді

<details>
<summary>Розгорнути відповіді</summary>

1. Пакетна фільтрація оцінює кожен пакет незалежно, без пам'яті про попередні пакети чи стан з'єднання. Stateful inspection відстежує повний життєвий цикл з'єднання (TCP handshake, послідовність номерів пакетів, стан сесії) — дозволяє правила типу «дозволити відповідь лише на запит, що вже був ініційований зсередини», що неможливо для stateless фільтра.
2. NGFW додає Deep Packet Inspection з розпізнаванням конкретного застосунку незалежно від порту (App-ID), інтеграцію з User Identity (Active Directory), Intrusion Prevention вбудовано, та Threat Intelligence з репутаційними базами, що оновлюються в реальному часі — на відміну від application layer firewall, що лише проксіює і фільтрує конкретні протоколи.
3. SSL/TLS Inspection дозволяє firewall бачити вміст зашифрованого трафіку (виявляти malware, ексфільтрацію), розшифровуючи і повторно шифруючи трафік (MITM за дизайном). Етична проблема: організація бачить весь приватний трафік співробітників, включно з банкінгом і медичними сайтами. Технічна проблема: ламає Certificate Pinning в мобільних застосунках і може порушувати end-to-end довіру TLS.
4. Принципи: Default Deny в кінці ruleset (все, що не дозволено явно — заборонено); найбільш специфічні правила зверху, загальні знизу; регулярний аудит для видалення застарілих/невикористовуваних правил; документування призначення кожного правила.
5. Сигнатурне виявлення порівнює трафік з базою ВІДОМИХ патернів атак — якщо атака нова (zero-day), сигнатури для неї ще не існує, тому вона не виявляється. Аномальне виявлення будує baseline нормальної поведінки і позначає будь-яке суттєве відхилення — нова атака, навіть невідома раніше, зазвичай поводиться інакше за нормальний трафік, тому потенційно виявляється.
6. NIDS аналізує мережевий трафік у конкретній точці мережі (на сегменті, через SPAN/TAP) — бачить весь трафік через цю точку, але не бачить що відбувається всередині хоста. HIDS встановлюється на конкретному хості — моніторить локальні логи, файлову систему, процеси — бачить деталі на рівні хоста, але не мережевий трафік в цілому. NIDS для виявлення мережевих атак і сканування; HIDS для виявлення компрометації конкретної системи.
7. Suricata підтримує multi-threading (використовує кілька CPU-ядер паралельно), на відміну від класичного однопотокового Snort — забезпечує вищу пропускну здатність для високошвидкісних мереж. Також Suricata має вбудовану глибшу підтримку аналізу протоколів і файлову екстракцію.
8. Zeek (Bro) не є сигнатурним IDS у класичному сенсі — він аналізує мережевий трафік на рівні протоколів і генерує детальні структуровані логи (HTTP-запити, DNS-резолюції, SSL-сертифікати, файлові транзакції) для подальшого аналізу або власних скриптів виявлення, а не лише зіставляє з готовими сигнатурами атак.
9. IPSec складається з AH (Authentication Header) — забезпечує автентифікацію і цілісність пакету без шифрування вмісту, та ESP (Encapsulating Security Payload) — забезпечує шифрування і опціонально автентифікацію. ESP використовується значно частіше, оскільки забезпечує конфіденційність.
10. WireGuard має значно меншу кодову базу (~4000 рядків коду проти десятків тисяч в IPSec-реалізаціях), що спрощує аудит безпеки і зменшує поверхню атаки через помилки реалізації. Використовує сучасну фіксовану криптографію (Curve25519, ChaCha20, Poly1305) без узгодження алгоритмів (cipher negotiation), що усуває цілий клас downgrade-атак, можливих в IPSec.
11. Split Tunneling: лише корпоративний трафік проходить через VPN-тунель, звичайний інтернет-трафік йде напряму без шифрування і без проходження через корпоративні засоби захисту (firewall, SWG). Ризик: компрометований пристрій з прямим доступом до інтернету може бути вектором атаки, що оминає корпоративний моніторинг; Full Tunneling спрямовує весь трафік через корпоративну інфраструктуру для повного контролю, ціною продуктивності.
12. Always-On VPN: VPN-з'єднання встановлюється автоматично і підтримується постійно, без можливості користувача його вимкнути, з моменту увімкнення пристрою. Вирішує проблему традиційного VPN: користувачі забувають увімкнути VPN або свідомо його вимикають для обходу корпоративного моніторингу — Always-On усуває цю людську змінну.
13. Мікросегментація ізолює на рівні окремого workload (контейнер, VM, навіть окремий процес), а не на рівні цілого мережевого сегмента як VLAN. VLAN типово об'єднує десятки чи сотні хостів в одну зону довіри; мікросегментація може створити унікальну політику для кожної пари workload, дозволяючи лише конкретно необхідний трафік.
14. Традиційний VPN надає доступ до всієї мережі (або великого сегмента) після одноразової автентифікації при підключенні. ZTNA надає доступ лише до конкретного запитуваного ресурсу/застосунку, з безперервною (а не одноразовою) перевіркою контексту (пристрій, ідентичність, поведінка) для кожного запиту — реалізує принцип найменших привілеїв на рівні мережевого доступу значно точніше, ніж VPN.
15. SD-WAN централізує політики безпеки і маршрутизації для розподілених офісів/хмарних з'єднань через програмне управління, дозволяючи застосовувати єдині security policies (шифрування, сегментація, інспекція трафіку) на всіх з'єднаннях незалежно від фізичного провайдера каналу, та динамічно маршрутизувати трафік через найбезпечніший/найоптимальніший шлях.
16. 802.1X — стандарт port-based мережевої автентифікації. Три сторони: Supplicant (пристрій, що намагається підключитись), Authenticator (мережевий комутатор/точка доступу, що контролює доступ до порту), Authentication Server (зазвичай RADIUS-сервер, що перевіряє credentials і приймає рішення).
17. MAB використовується для пристроїв, нездатних до повноцінної 802.1X автентифікації (принтери, IoT-пристрої, старе обладнання) — мережа дозволяє доступ на основі MAC-адреси пристрою замість повної автентифікації. Менш безпечний, бо MAC-адреси легко підробляються (spoofing) — зловмисник, що дізнався дозволену MAC-адресу, може видати себе за легітимний пристрій.
18. Posture Assessment перевіряє: чи увімкнений антивірус і чи актуальні його сигнатури, чи встановлені останні патчі безпеки ОС, чи увімкнено шифрування диска, чи відповідає пристрій корпоративним security policies — перш ніж надати повний мережевий доступ (або направити в карантинний VLAN для виправлення невідповідностей).
19. DNS спроєктований без вбудованої автентифікації відповідей — резолвер довіряє будь-якій відповіді, що виглядає легітимною (правильний transaction ID і UDP-порт відправника). Це уможливлює DNS Cache Poisoning/Spoofing: зловмисник підробляє відповідь, перенаправляючи жертву на шкідливий IP замість легітимного сервера.
20. DNSSEC додає криптографічні підписи до DNS-записів. Резолвер, що підтримує DNSSEC, перевіряє підпис відповіді відносно ланцюжка довіри від кореневої зони — підроблена відповідь без правильного приватного ключа власника домену не пройде перевірку підпису і буде відхилена, навіть якщо зловмисник вгадав transaction ID.
21. DoH/DoT шифрують DNS-запити, захищаючи приватність користувача від спостереження третіми сторонами (включно з власним ISP). Компроміс для організацій: корпоративні системи безпеки (DNS Filtering, виявлення DNS Tunneling, моніторинг для loC) традиційно покладаються на видимість DNS-трафіку у відкритому вигляді — DoH/DoT з зовнішніми резолверами (наприклад, у браузері) обходить цю видимість, ускладнюючи виявлення загроз через DNS.
22. DNS Tunneling кодує дані (команди C2 або вкрадену інформацію) усередині DNS-запитів (зазвичай у subdomain-частині, наприклад `base64data.attacker-domain.com`). Оскільки DNS-трафік зазвичай дозволений через firewall за замовчуванням (необхідний для функціонування мережі) і рідко глибоко інспектується, цей канал дозволяє обходити традиційні мережеві засоби контролю для ексфільтрації даних або підтримки командного каналу malware.
23. Forward proxy діє від імені клієнтів, приховуючи їх ідентичність від зовнішніх серверів і контролюючи вихідний трафік (типове корпоративне використання — фільтрація веб-доступу співробітників). Reverse proxy діє від імені серверів, приховуючи їх внутрішню архітектуру від зовнішніх клієнтів і захищаючи вхідний трафік (типове використання — захист вебзастосунків, балансування навантаження).
24. CASB надає видимість і контроль над використанням хмарних SaaS-застосунків (Shadow IT discovery, контроль доступу до конкретних хмарних сервісів, DLP для хмарних завантажень), доповнюючи SWG, що фокусується на загальному веб-трафіку — разом вони покривають як традиційний веб-перегляд, так і специфічну взаємодію з хмарними сервісами.
25. Certificate Pinning у мобільному застосунку перевіряє, що сертифікат сервера відповідає конкретному заздалегідь визначеному (pinned) сертифікату або публічному ключу. SWG з SSL Inspection підміняє оригінальний сертифікат сервера власним корпоративним сертифікатом для розшифрування трафіку (MITM за дизайном) — застосунок з Certificate Pinning виявляє цю підміну і відмовляється з'єднуватись, ламаючи функціональність застосунку.
26. WPA2 використовував 4-way handshake, вразливий до offline dictionary/brute-force атак на перехоплений handshake (особливо для слабких Pre-Shared Keys) і до атаки KRACK (Key Reinstallation Attack). WPA3 замінює це на SAE (Simultaneous Authentication of Equals, варіант Dragonfly handshake) — стійкий до offline-атак на пароль навіть при слабкому паролі, бо кожна спроба автентифікації потребує інтерактивної взаємодії з точкою доступу.
27. Rogue AP — несанкціонована точка доступу Wi-Fi, встановлена зловмисником (або необережним співробітником) у мережі або поруч з нею, часто з тим самим SSID що легітимна мережа (Evil Twin). Уможливлює MITM-атаку: жертва підключається до Rogue AP, вважаючи її легітимною, весь трафік жертви проходить через зловмисника.
28. Enterprise Wi-Fi (WPA2/WPA3-Enterprise) використовує 802.1X з індивідуальною автентифікацією кожного користувача через RADIUS — кожен співробітник має власні унікальні credentials, і доступ можна відкликати індивідуально без зміни паролю для всіх. Pre-Shared Key (WPA2-Personal) використовує один спільний пароль для всіх — компрометація або звільнення одного співробітника вимагає зміни паролю для абсолютно всіх пристроїв у мережі.
29. NetFlow збирає лише метадані про мережеві потоки (джерело, призначення, порти, протокол, обсяг даних, тривалість) — компактний обсяг даних, придатний для довготривалого зберігання і аналізу в масштабі всієї мережі. Packet Capture захоплює повний вміст кожного пакету, включно з payload — значно більший обсяг даних, але дозволяє детальний криміналістичний аналіз конкретного інциденту (що саме передавалось), тоді як NetFlow лише показує що комунікація відбувалась.
30. SPAN-порт — програмна функція комутатора, що дублює трафік на окремий моніторинговий порт; може втрачати пакети при високому навантаженні комутатора, оскільки конкурує з основною функцією комутації за ресурси. Апаратний TAP — фізичний пристрій, що пасивно копіює трафік на лінії без втручання в основний потік і без залежності від навантаження комутатора — забезпечує гарантоване захоплення 100% трафіку, включно з помилковими пакетами, які SPAN може фільтрувати.
31. Без baseline неможливо визначити, що є «аномальним» — будь-яка поведінкова система виявлення (як на рівні NIDS, так і на рівні мережевого моніторингу) потребує точки відліку «нормального» стану (типовий обсяг трафіку, типові протоколи, типовий час активності), відносно якої вимірюється відхилення. Без baseline система або генерує забагато false positives (все позначається як аномалія), або взагалі не може функціонувати.
32. NAC на вході перевіряє Posture пристрою при підключенні і поміщає його у відповідний сегмент (через інтеграцію з VLAN/мікросегментацією). NGFW контролює трафік між сегментами на рівні застосунків. NIDS моніторить трафік усередині сегментів на ознаки компрометації. Якщо пристрій, що пройшов початкову NAC-перевірку, пізніше скомпрометований і починає аномальну активність (сканування, C2-комунікація) — NIDS виявляє це, а NGFW з мікросегментацією обмежує можливе поширення на інші системи.
33. Шкідливе ПЗ після початкового зараження часто потребує зв'язку з C2-сервером для отримання команд або ексфільтрації даних. DNS Filtering блокує резолюцію відомих шкідливих доменів (з Threat Intelligence feeds), перериваючи цей канал ще до встановлення TCP/HTTP з'єднання — навіть якщо malware намагається зв'язатись з C2, DNS-запит до шкідливого домену просто не резолвиться в IP.
34. ZTNA не просто замінює механізм підключення (тунель замість VPN-тунелю) — він фундаментально змінює модель довіри: замість одноразової перевірки при вході з подальшою широкою довірою до всього мережевого сегменту, ZTNA забезпечує безперервну верифікацію контексту для кожного окремого запиту до кожного окремого ресурсу, реалізуючи принцип «ніколи не довіряй, завжди перевіряй» на рівні мережевого доступу, а не лише на рівні периметру.
35. Firewall на периметрі контролює трафік, що входить/виходить з усієї мережі організації загалом (north-south traffic) — захищає від зовнішніх загроз. Мікросегментація контролює трафік між internal workload (east-west traffic) — захищає від поширення загрози, що вже потрапила всередину периметру (lateral movement). Обидва шари необхідні: периметровий firewall не зупиняє поширення скомпрометованого внутрішнього хоста; мікросегментація без периметрового захисту не зупиняє початкове проникнення.
36. WPA3-Enterprise поєднує: SAE-подібний захист на рівні криптографічного протоколу (стійкість до offline-атак), 802.1X для індивідуальної автентифікації кожного користувача через RADIUS (відкличуваність доступу без зміни спільного паролю), і відповідно до Enterprise-вимог — 192-bit Security Suite для організацій з найвищими вимогами. Це комбінація найсильнішого доступного протоколу шифрування з найгранулярнішою моделлю автентифікації.
37. DNS Tunneling генерує характерний патерн мережевого трафіку: незвично велика кількість DNS-запитів до одного домену, незвично довгі subdomain-частини запитів, нетиповий обсяг даних через порт 53. NetFlow-аналіз, що відстежує обсяг і частоту з'єднань по протоколах, здатний виявити цю аномалію (наприклад, обсяг DNS-трафіку до конкретного хоста, що значно перевищує типовий baseline для легітимних DNS-запитів), навіть без глибокої інспекції вмісту кожного пакету.
38. Always-On VPN гарантує, що пристрій завжди підключений через захищений канал — але сам факт підключення не гарантує, що пристрій безпечний (актуальний антивірус, патчі, не скомпрометований). Posture Assessment перед наданням повного доступу додає другий незалежний рівень перевірки: підключення через VPN є необхідною, але недостатньою умовою — пристрій також має відповідати поточним вимогам безпеки перш ніж отримати доступ до чутливих ресурсів.
39. NGFW виконує SSL Inspection для трафіку, що проходить через мережевий периметр (весь трафік організації в цілому, незалежно від джерела запиту). SWG спеціалізується на вихідному веб-трафіку конкретних користувачів (часто з детальнішою політикою по користувачу/групі, включно з категоризацією веб-контенту). Обидва вирішують схожу задачу — видимість зашифрованого трафіку для виявлення загроз — але SWG зазвичай надає гранулярніший контроль саме для веб-специфічного use case, тоді як NGFW охоплює весь мережевий трафік загалом.
40. VLAN-сегментація — статична, заснована на мережевій топології (хто в якому сегменті) модель довіри: потрапивши у VLAN, пристрій довірений у межах усього сегмента. Zero Trust підхід виник як відповідь на обмеження цієї моделі: сучасні загрози (lateral movement, скомпрометовані легітимні credentials, BYOD, хмарні ресурси поза традиційним периметром) показали, що статична мережева позиція є ненадійним індикатором довіри. Організації рухаються до постійної верифікації контексту (хто, на якому пристрої, з яким рівнем ризику) для кожного доступу, незалежно від мережевого розташування — VLAN залишається корисним інструментом ізоляції, але більше не є самостійною основою моделі довіри.

</details>
