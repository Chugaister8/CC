# 13.7. Реєстр ризиків

## Від розрізнених оцінок до єдиного керованого документа

Розділи 13.3-13.6 дали методологію для ідентифікації активів, загроз та оцінки ризику (якісної й кількісної). **Реєстр ризиків (Risk Register)** — центральний документ, що зводить усі ці результати в єдину, живу, регулярно оновлювану таблицю, яка стає операційним інструментом управління, а не одноразовим звітом, що припадає пилом після завершення проєкту оцінки.

## Обов'язкові поля реєстру ризиків

За аналогією з попереднім блоком контексту завдання (таблиця Актив | Загроза | Вразливість | Ймовірність | Вплив | Рівень ризику | Контроль | Власник | Статус), розширений практичний реєстр включає:

| Поле | Призначення |
|---|---|
| ID ризику | Унікальний ідентифікатор (наприклад, RISK-001), наскрізний з розділу 13.4 |
| Дата ідентифікації | Коли ризик вперше внесено до реєстру |
| Актив | Прив'язка до реєстру активів (розділ 13.3) |
| Опис загрози/вразливості | Короткий опис сценарію (з таблиці розділу 13.4) |
| Ймовірність / Вплив | За якісною шкалою (розділ 13.5) |
| ALE (за наявності) | Кількісна оцінка за FAIR, якщо проводилась (розділ 13.6) |
| Рівень ризику (притаманний, inherent) | Рівень ризику **до** застосування додаткових контролів |
| Наявні контролі | Які контролі вже впроваджені й знижують ризик |
| Рівень ризику (залишковий, residual) | Рівень ризику **після** врахування наявних контролів |
| Обрана стратегія обробки | Уникнення / Зниження / Передача / Прийняття (розділ 13.8) |
| Заплановані дії | Конкретні кроки для реалізації обраної стратегії |
| Власник ризику | Особа, відповідальна за прийняття рішень щодо цього ризику |
| Термін виконання | Дедлайн для запланованих дій |
| Статус | Відкрито / В роботі / Закрито / Прийнято |
| Дата наступного перегляду | Коли ризик буде переоцінено повторно (розділ 13.11) |

## Притаманний ризик проти залишкового ризику

Ключова методологічна пара понять, яку часто плутають:

- **Inherent Risk (притаманний ризик)** — рівень ризику **без урахування** будь-яких контролів, ніби організація взагалі не має захисту. Корисний для розуміння теоретичного максимуму загрози.
- **Residual Risk (залишковий ризик)** — рівень ризику **після** врахування вже впроваджених контролів. Це те, з чим організація реально живе щодня, і саме залишковий ризик є предметом рішення про прийняття (розділ 13.8).

**Приклад:** RISK-001 (RCE через невиправлену CVE в API платежів, розділ 13.5) мав притаманний рівень «Критичний» (ймовірність 4, вплив 5). Після впровадження віртуального патчування (WAF-правило, Модуль 12, розділ 12.4) ймовірність знижується з 4 до 2 (Малоймовірна) — залишковий рівень стає «Середній». Реєстр фіксує обидва значення, оскільки вони відповідають на різні запитання: «наскільки небезпечна ця загроза сама по собі» (притаманний) проти «наскільки ми зараз захищені» (залишковий).

> **Міні-вправа 13.7.1:** У реєстрі ризиків компанії всі записи мають однаковий притаманний і залишковий рівень ризику для кожного сценарію. Що це найімовірніше свідчить, і чому це проблема з точки зору корисності реєстру?
>
> <details><summary>Відповідь</summary>
>
> Це найімовірніше означає, що поле «наявні контролі» фактично не заповнюється чи не враховується коректно при переоцінці — або організація дійсно не має впроваджених контролів (малоймовірно для більшості реальних активів), або (частіше) реєстр веде формально, без справжнього перерахунку залишкового ризику після впровадження заходів. Проблема: без чіткого розрізнення притаманного й залишкового ризику реєстр не може продемонструвати цінність уже зроблених інвестицій у контролі (немає видимого зниження ризику), і керівництво не отримує коректної картини поточного реального стану захисту, а лише теоретичного максимуму загрози.
> </details>

## Хто веде реєстр і як часто оновлюється

Реєстр ризиків — не одноразовий артефакт проєкту, а **живий документ**, що підтримується процесом безперервного управління ризиками (детальніше — розділ 13.11):

- **Власник реєстру (Risk Register Owner)** — зазвичай CISO (Chief Information Security Officer) чи еквівалентна роль, відповідальна за адміністрування самого документа, хоча власники окремих ризиків (Risk Owners) можуть бути розподілені по бізнес-підрозділах.
- **Регулярність оновлення** — новий сценарій ризику додається за подією (наприклад, результат Vulnerability Management чи пентесту, Модуль 12); плановий повний перегляд усіх записів — щоквартально чи щонапівроку, залежно від зрілості організації.
- **Тригери позапланового перегляду** — нова критична CVE в критичному активі, зміна бізнес-процесу (запуск нового продукту), інцидент безпеки, зміна регуляторних вимог.

## Реєстр ризиків як вхідні дані для звітності керівництву

Практична цінність реєстру — не лише операційний трекінг, а й основа для **executive summary** для ради директорів: агрегований підрахунок кількості ризиків за рівнем (наприклад, «3 Критичних, 8 Високих, 15 Середніх, наразі 2 Критичних ризики очікують затвердження бюджету на обробку»), тренд за квартал (чи зростає, чи знижується сумарна кількість Критичних ризиків), і найбільш значущі окремі ризики, що вимагають рішення керівництва прямо зараз.
